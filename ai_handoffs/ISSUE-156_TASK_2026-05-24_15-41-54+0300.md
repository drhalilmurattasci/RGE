`golden-projects/simple-scene`. The new regression must load the project
manifest as `rge_data::Project`, resolve its scene path, parse that file as
`rge_data::Scene`, build a `rge_kernel_ecs::World` with one ECS entity for each
parsed scene entity through a test-local identity-only helper, and assert that
the resulting world both contains those entities and advances its tick counters.
This dispatch must intentionally avoid a production scene loader or typed
component bridge.

## Planner Notes

This dispatch is the implementation follow-up to the ISSUE-154 preflight audit.
The audit found no existing direct `rge_data::Scene` to `rge_kernel_ecs::World`
bridge, but identified a smallest safe regression: keep the bridge private to a
new `rge-data` integration test and add `rge-kernel-ecs` only as a
dev-dependency. That gives the golden simple-scene fixture a real load+tick
regression without deciding where a production scene loader should live.

The intended helper shape is simple:

```rust
fn load_scene_into_world(scene: &rge_data::Scene) -> rge_kernel_ecs::World {
    let mut world = rge_kernel_ecs::World::new();
    for entity in &scene.entities {
        let id = rge_kernel_ecs::EntityId::from_ulid(entity.id.0);
        world.spawn_with_id(id);
    }
    world
}
```

Use the existing `golden_simple_scene_schema.rs` path-resolution pattern for
the project manifest and referenced scene. The new test may share ideas with
that file, but must not edit it. The current simple-scene fixture has one
parsed entity today; the regression should compare against `scene.entities`
rather than hard-coding that count so it remains correct if the fixture grows
in a future dispatch.

Under the orchestrator protocol, no `_REVIEW_` or `_CLOSEOUT_` packet is
expected from this TASK. The only new packet produced by execution should be
the `ISSUE-156_EXEC_*.md` report, plus its optional generated sidecar and the
