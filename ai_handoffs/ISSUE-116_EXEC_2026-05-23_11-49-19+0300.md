## 5-Question Render-Frame Perf Reconciliation Answer Block

Q1. What render-frame perf harness exists today, and what exact production path does it measure?

The harness exists at `crates/editor-shell/src/render_frame_e2e_perf.rs` and is gated as `#[cfg(test)] mod render_frame_e2e_perf;` from `crates/editor-shell/src/lib.rs:52-53` (no public re-export; the helpers it calls — `EditorShell::with_world_projection_graph`, `EditorShell::init_render_state_headless`, `EditorShell::acquire_depth_view`, `EditorShell::render_frame_to_target` — are all `pub(crate)`).

It measures the editor-shell substrate work that the production `crate::lifecycle::EditorShell::render_frame` (`crates/editor-shell/src/render_path.rs:319`-`441`) performs **AFTER** the winit-bound surface acquire (Phase B/C of production at `render_path.rs:336`-`360`) and **BEFORE** present (Phase G at `render_path.rs:431`-`439`), specifically:

1. `acquire_depth_view` (`render_path.rs:786`-`817`): `texture_pool.begin_frame()` + `buffer_pool.begin_frame()` + `build_resource_map(compiled, device, tex_pool, buf_pool)` + retrieval of the `DEPTH_RESOURCE_ID` texture and `create_view`. Same `DepthViewOutcome::Acquired` path Phase A of production exercises (`render_path.rs:324`-`333`).
2. `render_frame_to_target` (`render_path.rs:852`-`871`): creates a per-frame `wgpu::CommandEncoder` and calls `encode_main_pass` (`render_path.rs:900`-`1032`) which records one `RenderPass` (color attachment `LoadOp::Clear(DEFAULT_CLEAR)` + depth attachment `LoadOp::Clear(1.0)`, `StoreOp::Store`), sets the shared `LitMeshPipeline`, binds `gfx_camera` @0, `light` @1, per-mesh `material` @2, sets vertex / index buffers, and emits one `draw_indexed` per mesh (single-cuboid scene = 1 draw) plus the optional sub-ε highlight overlay (off in this harness because `highlight_index_buffer` is `None`); then `encoder.finish()` + `queue().submit(std::iter::once(...))`.

Winit surface acquire and present are explicitly **OUT** of scope: the harness allocates an offscreen `wgpu::Texture` color target (`render_frame_e2e_perf.rs:218`-`238`) sized 1024×768 with `wgpu::TextureFormat::Bgra8UnormSrgb` and `wgpu::TextureUsages::RENDER_ATTACHMENT`, calls `init_render_state_headless` (which uses `GfxContext::new_headless()` and skips the production Steps 1+3 winit `Window` / `SurfaceContext` build), and never invokes `surface.get_current_texture()` or `frame.present()` / `window.request_redraw()`. The module-level doc-comment at `render_frame_e2e_perf.rs:1`-`66` states this explicitly: "editor-shell encode/submit perf harness, **minus winit surface acquire/present**".

The production path that diverges between harness and production: production runs `encode_main_pass` + an optional Phase 9 egui pass against a single shared encoder before submit (`render_path.rs:382`-`429`); the harness re-uses the cuboid+overlay-only `render_frame_to_target` wrapper, which excludes the egui pass by design (`render_path.rs:840`-`871`, "Does NOT include the Phase 9 egui pass — the harness measures cuboid-render perf in isolation").
