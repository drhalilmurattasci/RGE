## Goal

Add focused test-only coverage for `AssetStoreCache::try_insert_*` backing-write failures in `crates/io-gltf/src/asset_store_cache.rs`. The uncovered material, animation, and skeleton fallible insert paths must prove that backing `put` errors surface as `GltfError::Cache` and leave the typed mirror unchanged. Add one recovery test that uses the same switchable backing across a failing call and a later successful call, proving the failure is transactional and the recovered handle is exactly the asset's `content_hash()` handle.

## Deliverables

- Add a `try_insert_material` negative test in the existing `asset_store_cache.rs` test module.
- Add a `try_insert_animation` negative test in the existing `asset_store_cache.rs` test module.
- Add a `try_insert_skeleton` negative test in the existing `asset_store_cache.rs` test module.

- Add one switchable-backing recovery test in the same test module. It must use one `AssetStoreCache` instance with the same backing toggled from failing to succeeding across `try_insert_*` calls, then assert the successful returned handle equals `asset.content_hash()`.

## Acceptance Criteria

- All new code is inside `crates/io-gltf/src/asset_store_cache.rs` under the existing `#[cfg(test)] mod tests`.
- There is a distinct negative test for each currently uncovered family: material, animation, and skeleton.
- The material failure test asserts `adapter.materials.len() == 0` and/or `adapter.get_material(&asset.content_hash()).is_none()` after the failed `try_insert_material` call.
- The animation failure test asserts `adapter.animations.len() == 0` and/or `adapter.get_animation(&clip.content_hash()).is_none()` after the failed `try_insert_animation` call.
- The skeleton failure test asserts `adapter.skeletons.len() == 0` and/or `adapter.get_skeleton(&skel.content_hash()).is_none()` after the failed `try_insert_skeleton` call.
- The recovery test does not rebuild the adapter between failure and success. It toggles one backing instance, such as with `Cell<bool>` or `AtomicBool`, from fail to succeed.
- The recovery test first observes a failed `try_insert_*` result and verifies the typed mirror remains empty or missing for the expected handle.
- The recovery test then toggles the same backing to success, calls the same fallible insert family again, and asserts the returned handle is exactly equal to the asset's `content_hash()` handle.
- The recovery test verifies the typed mirror contains the asset after the successful retry.
- No production behavior, public API, dependency, or Cargo metadata changes are made.
