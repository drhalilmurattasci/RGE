## Goal

Add one focused cad-projection smoke test for GitHub issue #23 proving that a
Cuboid projected through a topology-preserving Transform root still supports
B-Rep face lookup. This is a follow-up to Transform face-label preservation:
the test should exercise the projection layer's existing lazy face lookup and
must not change BRep identity derivation rules.

## Deliverables

- Exactly one new focused Rust test whose name contains `transform`.
- The test builds a `CadGraph` with `CuboidOp -> TransformOp`, sets the
  Transform node as the root, commits the graph, spawns a `CadProjection`
  entity for the Transform root, sets `BRepHandle.brep_owner`, ticks the
  projection, and proves lookup of an upstream Cuboid `BRepFaceId` still works.
- The proof may use either `CadProjection::face_resolves_in_projection` or
  `CadProjection::brep_face_id_for_triangle`, but it must compare against a
  `BRepFaceId` minted from the upstream Cuboid under the same `BRepOwnerId`.
- The test lives in only one place: either the existing
  `crates/cad-projection/src/lib.rs` test module or the optional integration
  test file allowed above. Do not duplicate the same smoke in both locations.

## Acceptance Criteria

- `cargo test -p rge-cad-projection transform -- --nocapture` exits 0.
- The filtered command above executes the new test because the test name
  contains `transform`.
- The new test fails if `TransformOp` projection loses face labels or if
  cad-projection face lookup no longer resolves through the Transform root to
  the upstream Cuboid face IDs.
- No production code, cad-core identity derivation, renderer/gfx code, Cargo
  metadata, docs, schemas, scripts, or unrelated tests are changed.

Cuboid graph, registering `BRepHandle`, setting `brep_owner`, ticking
projection, and comparing `BRepFaceId`s. Prefer reusing those local patterns.

root, not the upstream Cuboid node, so the smoke specifically covers lookup
through the projected Transform output.
