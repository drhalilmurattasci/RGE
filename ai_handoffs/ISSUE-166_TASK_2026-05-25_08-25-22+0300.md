## Goal

Make the four simple-scene component types from GitHub issue #166 directly attachable to the existing ECS `World`. ## Deliverables

- Add `rge-kernel-ecs = { workspace = true }` under `[dependencies]` in exactly these three manifests: `crates/components-spatial/Cargo.toml`, `crates/components-render/Cargo.toml`, and `crates/components-visibility/Cargo.toml`.
- Implement exactly four direct marker trait impls in the owning component crates: `impl rge_kernel_ecs::Component for Transform {}`, `impl rge_kernel_ecs::Component for Camera {}`, `impl rge_kernel_ecs::Component for Light {}`, and `impl rge_kernel_ecs::Component for Visibility {}`.
- Add at least one focused test per component crate proving the relevant component type or types can be inserted into a fresh `rge_kernel_ecs::World` and retrieved through the current ECS API. The render crate test may cover both `Camera` and `Light` in one test, or separate tests may be used.
- Preserve all existing serde/RON tests and public component APIs unless a minimal import path is needed for the direct ECS impls.
- Keep any `Cargo.lock` diff limited to the three dependency edges for `rge-components-spatial`, `rge-components-render`, and `rge-components-visibility` gaining `rge-kernel-ecs`.

## Acceptance Criteria

- Exactly the three named component manifests gain `rge-kernel-ecs` as a regular `[dependencies]` entry; it must not be added under `[dev-dependencies]`.

- `Transform`, `Camera`, `Light`, and `Visibility` each compile as `rge_kernel_ecs::Component` and can be inserted into a fresh `World` and retrieved with `world.entity(entity).unwrap().get::<T>()` or the equivalent current ECS API.
- `Cargo.lock`, if changed, contains no package additions, version bumps, checksum changes, source changes, or dependency churn other than the three permitted mechanical `rge-kernel-ecs` dependency-edge additions.
- No protected path is modified, and no new workspace member or loader crate is created.
