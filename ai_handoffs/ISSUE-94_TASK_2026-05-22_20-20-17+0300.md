## Goal

Implement GitHub issue #94 by adding an opt-in `rge-io-gltf` adapter that lets callers back the existing typed `crate::cache_stub::Cache` surface with `dyn rge_asset_store::Cache` byte storage. ## Acceptance Criteria

- `rge_io_gltf::MemoryCache` remains available, unchanged, and the default type used by all existing editor, loader, importer, exporter, and test call sites unless a caller explicitly opts into the new adapter.
- `rge_io_gltf::Cache` continues to refer to the existing local typed trait from `cache_stub.rs`; it is not replaced by, aliased to, or re-exported from `rge_asset_store::Cache`.
- The adapter is defined inside `rge-io-gltf` and bridges to `dyn rge_asset_store::Cache`; no trait or implementation changes are made in `crates/asset-store/**`.
- The existing public `crate::Cache` method signatures are byte-for-byte semantically preserved: no `Result` return is added, no handle type changes, no owned-return replacement for borrowed `get_*`, and no method removals or renames.
- The adapter's tests prove parity with `MemoryCache` for the covered asset family: inserting the same asset yields the same typed handle, getting that handle returns equivalent typed asset data, and re-inserting identical content does not create divergent handles.
- The adapter's tests prove asset-store-backed byte dedup: identical serialized/canonical bytes sent through `rge_asset_store::Cache::put` produce a single content-addressed asset-store entry or the same `AssetId`, and this is asserted through the asset-store backing path.
- The adapter's tests include a failing asset-store test double or equivalent failing backing cache and assert an `Err(GltfError::Cache(_))` from the adapter's fallible API.
- `GltfError::Cache` is used only for asset-store cache failures. Existing parse, schema, JSON, and ordinary file I/O error behavior must not be relabeled.
- `Cargo.lock` changes, if any, are limited to adding `rge-asset-store` to the existing `rge-io-gltf` package dependency list. No new third-party package, version, checksum, source, or unrelated dependency edge is acceptable.
- The diff is limited to `crates/io-gltf/src/asset_store_cache.rs`, `crates/io-gltf/src/lib.rs`, `crates/io-gltf/src/cache_stub.rs`, `crates/io-gltf/Cargo.toml`, `Cargo.lock`, and the required `ISSUE-94_EXEC_*.md` report.

## Planner Notes

Issue #92 established that `io-gltf`'s typed cache shape is not the same as `rge_asset_store::Cache`: `io-gltf` stores typed assets and returns borrowed `Option<&T>`, while asset-store stores fallible byte blobs and returns owned bytes. This dispatch intentionally avoids a wholesale replacement. A small typed mirror inside the adapter is acceptable if needed to satisfy the existing borrowed `get_*` lifetime shape, but the adapter must still exercise the asset-store backing path and tests must prove byte-level dedup/errors through that backing path.

Use the existing asset-family content-hash behavior as the compatibility anchor. Keep the bridge local to `asset_store_cache.rs`; avoid spreading serialization or cache policy into importer/exporter modules. The `cache_stub.rs` edit is only a doctrine/comment correction: the local typed trait remains the public contract for io-gltf, and the new asset-store bridge is an explicit opt-in for callers that want persistent byte backing.

Under the orchestrator protocol, no `_REVIEW_` or `_CLOSEOUT_` packet is expected from this TASK. The Executor should produce one `ISSUE-94_EXEC_*.md` report for this run.
