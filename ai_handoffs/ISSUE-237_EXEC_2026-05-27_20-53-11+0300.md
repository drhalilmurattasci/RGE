## What I Changed

### Source

- `Invoke-AiDispatchQueue.ps1`:
  - `Test-LooksLikePathToken` (around line 468): added a strict
    leading-dot repo path token branch
    (`^\.[A-Za-z0-9_-][A-Za-z0-9._-]*$`). The existing `/`, `*`, and
    short-extension accept gates are unchanged. The new branch accepts
    `.gitattributes`, `.gitignore`, `.env`, `.envrc`, `.editorconfig`,
    and similar bare leading-dot files without broadening to prose:
    bare `.`, `..`, `...`, whitespace-bearing strings, and ordinary
    prose tokens like `important` / `the` / `Write-DispatchLog` still
    return false. Slash-bearing dotfile paths such as
    `.cargo/config.toml` continue to clear the existing `/` gate.
  - `Fail` helper (around line 154): added a `Write-Output $Message`
    mirror before `exit 1` so terminal failures surface in captured
    queue output (stdout) even when the caller pipes only stdout to a
    log file. `[Console]::Error.WriteLine` is preserved so stderr
    output is unchanged for callers that do capture stderr.
  - Post-dispatch-log / pre-publish-decision window (around lines
    2319–2455): wrapped from the scope-guard timing trace through the
    publish-decision `Format-DispatchProgressComment` call in a single
    `try { ... } catch { Write-Output ...; throw }` block. The catch
    mirrors any non-`Fail` terminating error (raw `throw`, native
    command terminating error under `EAP=Stop`, or any other generalized
    exception) to stdout before re-throwing. The catch handler runs
    BEFORE the publish-decision `Send-DispatchProgressComment` so the
    failure is always visible in captured queue output, eliminating
    the silent-stall symptom seen on ISSUE-234.

## Open Questions for Reviewer

- The classifier accept pattern `^\.[A-Za-z0-9_-][A-Za-z0-9._-]*$`
  intentionally requires the first non-`.` character to NOT be a `.`,
  so `..` and `...` reject. Slash-bearing dotfile paths
  (`.cargo/config.toml`) clear the pre-existing `/` gate rather than
  this new pattern, so the new pattern itself is single-segment-only.
  Is that segmentation acceptable, or should the new pattern be
  extended to a multi-segment shape for consistency? The current
  approach keeps the new accept branch minimal and lets the existing
  `/` gate cover multi-segment cases without further surface.
- The Fail mirror to stdout is global (every Fail call site now
  emits to stdout in addition to stderr). I judged this benign
  because Fail is always the terminal error path and no code parses
  stdout for the Fail message specifically, but if a future caller
  prefers to keep error text on stderr only, the mirror would need
  to be gated by a parameter or moved to a scoped wrapper.
- The try/catch around the post-log/pre-publish-decision window
  preserves the existing indentation of the wrapped body (per the
  inline comment) to keep the diff review-friendly. PowerShell does
  not depend on indentation, so this is purely a review cost vs.
  noise tradeoff. Happy to re-indent the ~120-line body in a
  correction round if preferred.
