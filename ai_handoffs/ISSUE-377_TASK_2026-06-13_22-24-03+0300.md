remain visible in diagnostics and display/introspection lookups, but keyboard
execution must not fire a shortcut while `ResolveResult::conflicts` contains
that shortcut. Keep the change inside the existing `editor-ui` menu/shortcut
resolution substrate, with focused tests and only minimal editor-shell
accelerator parity comments/tests if needed.

## Deliverables

- Update the `ResolveResult` execution resolver so
  `enabled_command_for_shortcut` returns `None` for any shortcut present in
  `ResolveResult::conflicts`, even when the first-registered conflicting entry
  is enabled.
- Preserve `ResolveResult::command_for_shortcut` and
  `AcceleratorTable::resolve` first-registered winner behavior for display,
  diagnostics, and compatibility.
- Preserve hidden and predicate-filtered entries releasing their shortcut slots,
  and preserve disabled-but-visible entries as display bindings that do not
  execute.
- Add or revise focused tests covering conflicted suppression, unconflicted
  execution, disabled suppression, hidden/predicate filtering, and deterministic
  diagnostics order.
- Update editor-shell accelerator comments or parity assertions only if needed
  to keep them accurate with the already-present `enabled_command_for_shortcut`
  execution path.
- Update task/status/history docs for task 123 completion without appending any
  task 124.
- Write the EXEC report for this dispatch.

## Acceptance Criteria

- For a live conflicted shortcut, `ResolveResult::enabled_command_for_shortcut`
  returns `None`.
- For that same conflicted shortcut, `ResolveResult::command_for_shortcut` still
  returns the first-registered command and `AcceleratorTable::resolve` still
  returns the first-registered entry id.
- Existing conflict diagnostics still report all conflicting entry ids in
  deterministic registration order, and multi-conflict diagnostics remain in
  deterministic shortcut-display order.
- An unconflicted enabled shortcut still executes through
  `enabled_command_for_shortcut`.
- A disabled-but-visible shortcut still appears as a binding through
  `command_for_shortcut` but returns `None` through
  `enabled_command_for_shortcut`.
- Hidden entries and predicate-filtered entries do not occupy accelerator slots;
  a visible entry can claim the same shortcut without being suppressed by a
  hidden/predicate-filtered entry.
- The editor-shell keyboard path remains routed through
  `keycode_to_shortcut` plus `ResolveResult::enabled_command_for_shortcut`;
  no new route, queue, handoff, command sink, or plugin executor is introduced.
- `.ai/dispatch.tasks.md` records task 123 completion and contains no `124.`
  task heading.
- `git diff --name-only` contains only MAY-edit files plus this dispatch's
  generated EXEC/run/log artifacts.

## Planner Notes

Task 122 reached a policy boundary and the delegated-human decision selected
the smallest keybinding-conflict policy slice: live conflicts are diagnostic
data and display/introspection metadata, but not executable keyboard bindings.
The shell already executes canonical menu accelerators through
`enabled_command_for_shortcut`; prefer making that resolver conflict-aware over
introducing a new shell-side route or a second shortcut policy surface.
