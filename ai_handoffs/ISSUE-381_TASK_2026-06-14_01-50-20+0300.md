## Acceptance Criteria

- An enabled shortcut-help row whose displayed shortcut matches a
  `ProjectedMainMenu.conflicts[*].shortcut` value exposes peer entry ids in the
  same order as that conflict's `entries` vector.
- Unconflicted enabled rows expose no conflict peer detail.
- Disabled rows expose no conflict peer detail. A disabled row whose displayed
  shortcut appears in `ProjectedMainMenu.conflicts` remains `Disabled`, not
  `Conflicted`.
- Projected plugin rows can expose conflict peer ids when the existing projected
  plugin row is enabled and its displayed shortcut matches an existing projected
  conflict. The implementation must work from the already-projected menu data,
  not from registry/plugin re-resolution.
- The existing Keyboard Shortcuts help State labels remain distinct:
  `Enabled`, `Disabled`, and `Conflicted`.
- Existing shortcut-conflict diagnostics in `shortcut_conflicts.rs` continue to
  expose all conflicting entry ids in their current order and remain sourced from
  `ProjectedMainMenu.conflicts`.
- Shortcut execution, menu clicks, command-palette activation, conflict
  diagnostics, remapping/persistence, plugin runtime/discovery/loading,
  clipboard, CAD/CommandBus, camera/navigation, Cargo, workflow, and automation
  behavior remain unchanged.
- `git diff --name-only` contains only MAY-edit files plus this dispatch's
  generated EXEC/run/log artifacts.
- `.ai/dispatch.tasks.md` has no task 128 heading after the change.

they are not fatal and are not remapped here. Prefer a small local data-shape
change in `shortcut_help.rs`, such as an empty-by-default peer-id vector/option
on `ShortcutHelpRow`, populated only for enabled rows whose displayed shortcut

`ProjectedMainMenu` tests are the safest way to pin plugin rows and
disabled-row behavior without widening source scope.
