## What I Changed

### Source

- `Wait-GitHubActions.ps1`:
  - `Test-PrSideAcceptanceConfig` (≈ line 320): dropped the `-Expected`
    parameter and the "every accepted name must already appear in
    `-WorkflowName`" rule. The validator now only enforces the
    `-PullRequest <PR#>` requirement and returns the deduplicated accepted
    list. Updated the function doc-comment so callers understand the new
    contract — accepted names are unioned into the expected set by the
    waiter itself.
  - `Invoke-WaitForGitHubActions` (≈ line 384): moved the
    `Test-PrSideAcceptanceConfig` call to run before `$expected` is
    computed, then built `$expected` from the union of `-WorkflowName` and
    the validator's returned accepted list (deduplicated, nulls dropped).
    This is what makes `-PullRequest 240 -AcceptPrSideCheck CodeQL` work
    without `CodeQL` having to appear in `-WorkflowName`. Default strict
    behaviour for tracked workflows is unchanged: when `-AcceptPrSideCheck`
    is empty, the validator returns `@()` and the expected set collapses
    back to `-WorkflowName` exactly as before.
  - `Resolve-PrSideCheckEvidence` (≈ line 275): when normalising a
    StatusContext entry (`-not $conclusion -and $check.state`), the
    classifier now inspects the state value. `PENDING` and `EXPECTED` (the
    two unresolved values in the commit-status state vocabulary) return
    `State = 'pending'` immediately, so the waiter keeps polling and exits
    `2` on timeout. Other state values still fall through to the existing
    "COMPLETED" branch, where `SUCCESS` is the only `pass` and everything
    else is `fail`.
  - `.PARAMETER AcceptPrSideCheck` and the second `.EXAMPLE` block:
    rewritten to match the new contract. The example now shows
    `-PullRequest 240 -AcceptPrSideCheck CodeQL` without listing CodeQL in
    `-WorkflowName`.

### Tests

- `tools/dispatch-tests/Issue241-WaitGitHubActions-PrSideCheck.Tests.ps1`
  — updated and extended. The waiter is still dot-sourced through
  `RGE_WAIT_GITHUB_ACTIONS_SKIP_MAIN`, and all end-to-end tests inject
  `Clock`, `Sleeper`, `RunListProvider`, and `PrRollupProvider`; no live
  `gh` / `git` / filesystem mutation. Pester reports 36 passed / 0 failed
  / 0 skipped (was 31; +5 net).

  - Classifier (`Describe 'ISSUE-241 Resolve-PrSideCheckEvidence ...'`):
    - Added `returns pending for a StatusContext entry with state
      <State>` parametrised over `PENDING` and `EXPECTED`. Asserts both
      `State = pending` and that the raw `Conclusion` is echoed back.
    - Added `still returns fail for a StatusContext entry with state
      FAILURE` to pin that terminal failing StatusContext states are
      still terminal failures.

  - Validator (`Describe 'ISSUE-241 Test-PrSideAcceptanceConfig ...'`):
    - Dropped `-Expected` from the empty-input, missing-`PullRequest`, and
      dedup test invocations to match the new signature.
    - Replaced the prior `rejects an accepted name that is not in
      -WorkflowName` test with the inverse: `accepts a name even when it
      is not pre-listed in -WorkflowName`, which proves the validator no
      longer enforces that link.
    - Comment block at the top of the file was updated (rule 6) to
      document the new validator contract, and a new rule 7 was added
      pinning the StatusContext PENDING / EXPECTED behaviour.

  - End-to-end (`Describe 'ISSUE-241 Invoke-WaitForGitHubActions ...'`):
    - Added `accepts -PullRequest 240 -AcceptPrSideCheck CodeQL without
      duplicating CodeQL in -WorkflowName`: passes `-WorkflowName @()`, a
      matching-SHA SUCCESS PR rollup, and asserts exit 0. This is the
      direct acceptance test the correction packet asked for.
    - Added `keeps polling and times out (exit 2) when the PR-side
      StatusContext is still PENDING`: passes a StatusContext entry with
      `state = PENDING` and a fast-forward sleeper, and asserts exit 2 —
      confirming the unresolved-state behaviour at the wait loop level.
