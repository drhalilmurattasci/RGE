## Goal

Add the smallest editor-shell-owned extension-command executor seam beyond the
existing captured extension command FIFO. `Command::Custom` and
`Command::Plugin` menu activations already reach
`EditorShell::route_menu_command` and are captured in editor-shell state; this
dispatch must add an injectable shell seam that can process those captured
extension commands without wiring real plugin runtime, discovery, loading, WASM,
capability, async, or sandbox infrastructure.

## Deliverables

- An editor-shell-owned extension-command executor seam with an injectable
  handler used only for captured `Command::Custom` and `Command::Plugin`
  activations.
- FIFO processing from the existing extension command capture path into the
  injected handler, without delivering representative core commands to that
  handler.
- Explicit, observable, non-fatal behavior for missing handlers, handler
  failures, and unhandled extension commands.
- Focused editor-shell tests using a synthetic handler for FIFO delivery,
  core-command non-delivery, no-handler behavior, and failure/unhandled
  behavior.
- Status/task bookkeeping that records this dispatch as an editor-shell seam
  only, with no real plugin runtime, discovery, or loading wired.

## Acceptance Criteria

- A configured synthetic handler receives `Command::Plugin` and
  `Command::Custom` activations in FIFO order after menu routing.
- Core commands, including representative commands such as Save and Toggle
  Command Palette, are not delivered to the extension executor seam and keep
  their existing document/menu behavior.
- No-handler behavior is covered and keeps captured extension activations
  observable rather than silently dropping them.
- Handler failure or unhandled results are non-fatal, do not invoke document
  handlers, and do not prevent later extension commands from being processed.
- The seam accepts only extension commands captured by
  `EditorShell::route_menu_command`; it does not introduce generalized registry
  execution or host-to-shell FIFO replacement.
- No plugin host, plugin discovery, plugin loading, WASM runtime, capability
  manifest, async executor, sandbox integration, Cargo metadata, workflow, or
  architecture-lint change is introduced.

## Constraints / Non-Goals

- Do not implement real plugin runtime, plugin discovery, plugin loading, WASM
  execution, manifest capability checks, async execution, or sandboxing.
- Do not edit editor UI command definitions, editor-egui-host menu
  projection/registration behavior, host-to-shell FIFO semantics, plugin/runtime
  crates, kernel plugin-host code, Cargo metadata, workflows, dispatch
  automation, schemas, or architecture lints.
- Do not solve generalized command registry execution, keybinding editor
  behavior, conflict fatality policy, OS clipboard, typed clipboard, CAD
  graph/projection mutation, deletion/duplication, undo integration, dirty-state
  quit prompts, or broader camera UI.

- Do not name the stale route-menu owner in final docs or task bookkeeping;

  `route_menu_command` reference.
