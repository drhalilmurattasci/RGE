## What I Changed

### Source

- `Wait-GitHubActions.ps1`:
  - Added two new script parameters: `[int]PullRequest` and
    `[string[]]AcceptPrSideCheck`. The opt-in is strictly off by default;
    both parameters must be supplied to enable PR-side resolution.
  - Extracted the main loop into a callable function
    `Invoke-WaitForGitHubActions` that returns the intended exit code
    (0/1/2) instead of calling `exit`. The function accepts injectable
    `-Clock`, `-Sleeper`, `-RunListProvider`, and `-PrRollupProvider`
    scriptblocks so deterministic tests can drive it without shelling
    out to `gh` or sleeping for real.
  - Added a dot-source seam: the script returns early when the
    environment variable `RGE_WAIT_GITHUB_ACTIONS_SKIP_MAIN` is set, so
    Pester can dot-source the script to bring helpers into scope
    without running the main flow.
  - Added a strict PR-side classifier `Resolve-PrSideCheckEvidence` that
    returns one of `pass | fail | pending | missing | sha-mismatch`.
    Rules:
    - Accept only when the PR rollup's `headRefOid` equals `-Commit`.
    - Accept only an explicit `SUCCESS` conclusion as `pass`.
    - Treat FAILURE / CANCELLED / TIMED_OUT / SKIPPED / NEUTRAL /
      ACTION_REQUIRED / STARTUP_FAILURE / unknown conclusion as `fail`
      (terminal but not a pass).
    - Treat in-progress / queued status as `pending`.
    - Treat missing-from-rollup, null rollup, or empty rollup as
      `missing`.
    - Normalises both the `name` + `conclusion` (CheckRun) and
      `context` + `state` (StatusContext) rollup shapes.
  - Added `Test-PrSideAcceptanceConfig`, a fail-fast validator that
    rejects `-AcceptPrSideCheck` without `-PullRequest` and rejects any
    accepted name not also present in `-WorkflowName`.
  - Added `Get-PrSideCheckRollup`, a thin `gh pr view` JSON wrapper
    used when no `-PrRollupProvider` is injected.
  - PR-side resolution fires only for rows whose `gh run list` result
    is `missing` and whose name is in the opt-in list, so the default
    strict semantic for tracked workflows is preserved bit-for-bit.
  - Failed-row classification now distinguishes by `Source`: run-list
    rows still accept `success`/`skipped`/`neutral`; PR-side rows
    accept only `success`.

## Open Questions for Reviewer

- `-AcceptPrSideCheck` requires every accepted name to also appear in
  `-WorkflowName`. This keeps the opt-in surface narrow but means a
  caller who wants the org/default CodeQL check waited on must list
  it twice (once in `-WorkflowName`, once in `-AcceptPrSideCheck`).
  I considered making `-AcceptPrSideCheck` implicitly add to the
  expected set, but that would make a single `-AcceptPrSideCheck`
  invocation widen the wait target - confusing under the strict
  contract. Confirm the explicit "list twice" design is acceptable.
- PR-side `SKIPPED` is classified as `fail` (exit 1) rather than
  ignored, in line with the TASK packet's "must not be accepted as
  pass" clause that explicitly names skipped. If reviewers prefer
  `SKIPPED` to keep polling (treat as `missing` -> eventually exit 2),
  the change is a one-line tweak in `Resolve-PrSideCheckEvidence`.
