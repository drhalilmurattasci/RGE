## Goal

Implement stable B-Rep edge IDs for `SweepOp` and route graph-level Sweep edge resolution through that direct provider. The implementation must reuse the existing Sweep face-ID substrate in `crates/cad-core/src/operators/sweep.rs` and derive every edge with `BRepEdgeId::for_face_pair`. This closes GitHub issue #19 without changing Sweep geometry, face-label ordering, face-ID derivation, or unrelated operator behavior.

## Deliverables

- Add `impl BRepEdgeProvider for SweepOp` in `crates/cad-core/src/operators/sweep.rs`.
- Use `self.brep_face_ids(owner)` to obtain the canonical Sweep face IDs, then derive edge IDs only through `BRepEdgeId::for_face_pair(face_a, face_b, 0)`.
- Mirror the existing face provider's defensive `path_segment_count = m.saturating_sub(1)` posture so a hypothetical degenerate public-field state cannot panic during edge indexing. If `s == 0`, return an empty edge list rather than indexing nonexistent side faces.
- Emit Sweep edge IDs in this canonical order, where `n = profile.len()` and `s = path.len().saturating_sub(1)`:
  - First-cap perimeter: `n` edges, `FirstCap` with `Side(0, i)`.
  - Last-cap perimeter: `n` edges, `LastCap` with `Side(s - 1, i)`.
  - Segment side seams: `n * s` edges, `Side(k, i)` with `Side(k, (i + 1) % n)` for each path segment `k`.
  - Interior ring edges: `n * (s - 1)` edges, `Side(k, i)` with `Side(k + 1, i)` for each adjacent segment pair.
- The direct provider edge count must be `n * (2 * s + 1)` for valid Sweep paths, with `local_ordinal = 0` for every edge.
- Update `crates/cad-core/src/topology/edge_resolve.rs` so `OperatorNode::Sweep(op)` returns `Ok(op.brep_edge_ids(owner))` instead of falling through to `BRepResolveError::TopologyChangingOperator`.
- Update stale prose in `crates/cad-core/src/topology/edge_resolve.rs` that currently classifies Sweep as topology-changing for edge resolution; Boolean must remain in that category.
- Add focused unit tests covering both 2-point and multi-segment paths:
  - edge count,
  - canonical face-pair adjacency and output order,
  - rebuild stability for same topology with changed profile/path coordinates,
  - resolver parity with direct `BRepEdgeProvider` output.
- Update existing edge-resolver tests so Boolean remains topology-changing, while Sweep resolves successfully.
- Update `crates/cad-core/tests/brep_edge_chain_resolution_smoke.rs`:
  - replace `sweep_returns_topology_changing_error` with a test asserting `brep_edge_ids_for_node` returns `Ok` and exactly equals `sweep.brep_edge_ids(owner)`;
  - keep `boolean_returns_topology_changing_error` unchanged except for any formatting needed nearby;
  - correct the module-doc numbered list so item #7 describes Sweep direct resolution rather than `TopologyChangingOperator`.
- Normalize LF line endings for exactly the MAY-edit files that the live rustfmt gate flags for `Incorrect newline style`; currently expected from prior gate review: `sweep.rs`, `edge_resolve.rs`, `edge_id.rs`, `provider.rs`, and `tests/brep_edge_chain_resolution_smoke.rs`.

## Acceptance Criteria

- `SweepOp` implements `BRepEdgeProvider` directly and compiles without changing public trait signatures.
- For a square profile over a 2-point path, Sweep returns `12` edge IDs and the expected `FirstCap`/`LastCap`/side-seam adjacency order.
- For a triangle profile over a 3-point path, Sweep returns `15` edge IDs and tests pin at least one first-cap edge, one last-cap edge, one segment side seam from each segment, and one interior ring edge.
- Rebuilding a Sweep with the same `profile_count` and `path_segment_count` but different profile/path coordinates yields byte-identical edge IDs for the same owner.
- Resolving `OperatorNode::Sweep` through `brep_edge_ids_for_node` returns `Ok` and exactly equals `sweep.brep_edge_ids(owner)` for both 2-point and multi-segment paths.
- Existing Boolean edge resolution still returns `BRepResolveError::TopologyChangingOperator { kind: OpKind::Boolean }`.
- The integration smoke test in `crates/cad-core/tests/brep_edge_chain_resolution_smoke.rs` no longer expects Sweep to be topology-changing and is covered by the non-`--lib` crate test gate.
- No Sweep mesh geometry, triangle indices, face labels, face-tag discriminants, face-ID derivation, or edge-ID derivation changes.
- Prior verification feedback is addressed: the stale `transform.rs` newline exception is removed, LF normalization is permitted only for the explicitly listed MAY-edit files, and `transform.rs` is not changed.

## Constraints / Non-Goals

- Do not change Sweep's v0 path semantics, monotonic-Z restriction, winding correction, cap/side triangle ordering, or per-triangle `TopologyFaceId` labels.
- Do not change `SweepFaceTag` fields, discriminants, serialization, or `BRepFaceId::for_sweep_face`.
- Do not introduce coordinate-aware edge identity. Stable Sweep edge IDs are topology-derived from face IDs only.
- Do not add a new topology trait, new resolver error variant, or new local ordinal scheme.
- Do not implement edge IDs for Boolean or any operator other than Sweep in this dispatch.
- Do not broaden fillet, transform, cuboid, extrude, revolve, or loft behavior.
