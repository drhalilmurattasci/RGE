## Classification Rules

- Use `pressure_deferred` only when current evidence shows the cavity's v0
  non-goals or admission gate still hold and no current consumer path requires
  that cavity to grow now.
- Use `newly_triggered` only when current code, tests, manifests, or current
  docs show a concrete present consumer pressure that requires capability
  explicitly excluded by the cavity's v0 non-goals or admission gate. Roadmap
  ambition, old closeout text, crate names, open-ended comments, or ordinary
  future extensibility are not enough.
- For `kernel/shared`, `newly_triggered` requires observed duplication of a
  type or function across at least two implemented kernel crates, plus
  evidence that neither owning kernel crate is the natural home. Convenience,
  import cleanup, or anticipated future duplication remains
  `pressure_deferred`.
- For `kernel/asset-streaming`, `newly_triggered` requires current consumer
  pressure for a residency manager, hysteresis, predictive prefetch, memory
  budget enforcement, actual I/O coupling, asset ID integration, GPU upload
  coupling, or another behavior that the v0 crate explicitly excludes.
- For `kernel/io-scheduler`, `newly_triggered` requires current consumer
  pressure for actual I/O driver dispatch, async runtime integration,
  executor semantics, DAG scheduling, GPU upload semantics, distributed
  coordination, reactive scheduling, or cross-cavity priority coupling that
  the v0 crate explicitly excludes.
- For `kernel/job-system`, `newly_triggered` requires current consumer pressure
  for a work-stealing pool, actual work execution, closure storage,
  cancellation, task graph scheduling, async runtime integration, priority
  inversion handling, thread affinity, or another execution feature that the
  v0 crate explicitly excludes.
- For `kernel/asset-view`, `newly_triggered` requires current consumer pressure
  for WASM linear-memory mapping, unsafe zero-copy slice exposure, real buffer
  or allocation ownership, asset integration, residency coupling, GPU upload
  semantics, or another mapping feature that the v0 crate explicitly excludes.
- If more than one cavity is `newly_triggered`, still name exactly one
  smallest safe follow-up unless the evidence cannot justify ordering. If
  ordering requires product, architecture, scheduler, unsafe-boundary,
  runtime, or roadmap judgment, return `NEEDS_HUMAN`.
