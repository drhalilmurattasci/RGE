## Acceptance Criteria

- Both new accessors are methods on the already-exported `ResolveResult`; no
  `mod.rs` export, public type, enum, field, serde, settings, persistence,
  host/shell, plugin runtime, CAD, Cargo, workflow, schema, or automation change
  is introduced.
- The accessors are read-only over the existing private `shortcut_commands`
  field. Do not add a new field, secondary index, cached vector, global state, or
  resolve-time side table.
- `bindings()` returns all resolved shortcut-command pairs in a documented
  stable order and does not leak raw `HashMap` iteration order.
- `shortcut_for_command(&Command)` uses the same documented stable order to
  select one shortcut when multiple shortcuts resolve to the same command; the
  many-shortcuts-to-one-command rule is documented in the method docs.
- Resolve behavior is unchanged: no change to `resolve`,
  `resolve_with_keybinding_overrides`, override application, conflict
  detection/suppression, `keybinding_diagnostics`, `AcceleratorTable`, default
  menu entries, labels, order, predicates, enablement predicates, shortcut
  values, or passive shortcut hints.
- Tests in `crates/editor-ui/tests/menus_ordering.rs` cover all of these:
  a bound command returns its shortcut; an unbound command returns `None`;
  `bindings()` enumerates every effective pair in the documented stable order;
  remap and unbind overrides are reflected by both accessors; repeated resolves
  produce identical accessor results; the default menu still has exactly 19
  executable accelerators and zero conflicts; and no prohibited diff exists.
- `.ai/dispatch.tasks.md` has exactly one new task 177 audit block appended by
  the final step, and no live `NEEDS_HUMAN_RECORDED` marker is added unless the
  self-rearm requirement itself says it is permitted.
