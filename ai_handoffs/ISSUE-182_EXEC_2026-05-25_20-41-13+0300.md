## What I Changed

### Source

- `Invoke-AiDispatchLoop.ps1` (top-level param block, ~line 84): added
  `[ValidateRange(0, 7200)] [int]$CodexStallThresholdSec = 300` alongside the
  existing timeout parameters. A value of `0` disables the watchdog and
  leaves only `ModelTimeoutSec` in effect.
- `Invoke-AiDispatchLoop.ps1::Invoke-WithTimeout` (~line 181): extended the
  signature with optional `[ValidateRange(0, 7200)] [int]$StallThresholdSec = 0`
  and `[ValidateRange(1, 600)] [int]$PollIntervalSec = 5`. The PowerShell
  5.1 range validation prevents tight polling loops without runtime
  clamping; the additional 50 ms floor inside the loop guards against
  sub-second deadline overshoot windows.
- `Invoke-AiDispatchLoop.ps1::Invoke-WithTimeout` body (~lines 233-310):
  preserved the legacy branch verbatim when `StallThresholdSec -le 0` and
  added a parallel stall-watchdog branch. The watchdog:
  - polls `proc.HasExited`,
  - checks the hard timeout against `DateTime.UtcNow` (legacy semantics:
    `Code=124`, `TimedOut=$true`),
  - arms only after `OutFile` exists and `Length -gt 0`,
  - tracks `OutFile.Length` and `LastWriteTimeUtc`, refreshing the stall
    deadline only when one advances,
  - on stall: `taskkill /T /F /PID $proc.Id`, `Code=125`, `TimedOut=$true`,
    `Stalled=$true`,
  - caps each `Start-Sleep -Milliseconds` by the nearest of the two
    deadlines so the kill is prompt.
- `Invoke-AiDispatchLoop.ps1::Invoke-WithTimeout` return shape: legacy
  branch still returns `{ Code; TimedOut }` unchanged; only the stall
  branch adds the `Stalled` field. This keeps existing non-Codex callers
  (`Invoke-ClaudeMarker`, `Test-ClaudeCliReady`, `Invoke-Verification`,
  `Invoke-GitCapture` is unaffected because it does not call this helper)
  observing the identical object shape.
- `Invoke-AiDispatchLoop.ps1::Invoke-CodexPrompt` (~line 266): passes
  `-StallThresholdSec $CodexStallThresholdSec` only at this call site, and
  emits a stall-specific failure message naming the threshold and log
  path when `r.Stalled` is `$true`. The non-stall hard-timeout message and
  the non-zero exit branch are untouched.

## Open Questions for Reviewer

- The `Stalled` field is only present on the result object when the
  caller opts into the watchdog (legacy callers continue to see
  `{ Code; TimedOut }` only). `Invoke-CodexPrompt` reads it through
  `r.PSObject.Properties['Stalled']` to keep the legacy branch return
  shape strictly identical; please confirm that mechanism is acceptable
  vs. always emitting `Stalled=$false` in the legacy branch.

  near-deadline iteration does not bound the kill into the next second.
  This is well above a tight-loop threshold but well below the
  user-visible `PollIntervalSec` default of 5 s.
