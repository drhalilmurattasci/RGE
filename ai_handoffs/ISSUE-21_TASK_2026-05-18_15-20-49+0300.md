## Acceptance Criteria

- The new test file compiles as an integration test in the existing `rge-cad-projection` crate without any Cargo manifest changes.
- The positive smoke uses existing public APIs only: `CadGraph`, `OperatorNode::Sweep`, `Polygon2D`, `Polyline3D`, `SweepOp`, `BRepProvider::brep_face_ids`, `BRepHandle`, `CadProjection`, and `World`.
- For the square profile with 3 path points, the projected mesh has exactly 20 triangles: `2 * 4 * 2 + 2 * (4 - 2)`.
- The direct provider comparison calls `SweepOp::new(unit_square(), z_path(&[0.0, 1.0, 2.0])).brep_face_ids(TEST_OWNER)` and observes exactly 10 face IDs: first cap, last cap, then `4 * 2` side faces.
- Every triangle index `0..20` resolves through `CadProjection::brep_face_id_for_triangle(entity, tri, &world, graph.graph())` to the corresponding direct `SweepOp::brep_face_ids(TEST_OWNER)` ID.
- Canonical order is asserted explicitly:
  - triangles `0..2` resolve to direct face ID index `0` (`FirstCap`);
  - triangles `2..4` resolve to direct face ID index `1` (`LastCap`);
  - for each `segment_index in 0..2` and `edge_index in 0..4`, triangles `4 + 2 * (segment_index * 4 + edge_index)` and `4 + 2 * (segment_index * 4 + edge_index) + 1` resolve to direct face ID index `2 + segment_index * 4 + edge_index`.
- An out-of-bounds lookup, such as triangle index `99`, returns `None`.
- After setting the entity's `BRepHandle.brep_owner` to `None`, a valid triangle lookup returns `None`.
- No production code, provider code, Sweep geometry, renderer/gfx code, kernel code, docs, scripts, manifests, or unrelated tests are changed.
