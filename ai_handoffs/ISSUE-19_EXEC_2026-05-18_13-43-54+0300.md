## What I Changed

### Source

- `crates/cad-core/src/operators/sweep.rs`: Added `impl BRepEdgeProvider for
  SweepOp` directly after the existing `impl BRepProvider for SweepOp`. The
  provider obtains the canonical Sweep face IDs via `self.brep_face_ids(owner)`
  and derives every edge purely through `BRepEdgeId::for_face_pair(face_a,
  face_b, 0)` — no coordinate input, `local_ordinal = 0` on every edge. It
  mirrors the face provider's defensive posture: `s =
  self.path.len().saturating_sub(1)`, and `s == 0` returns an empty edge list
  rather than indexing nonexistent side faces. Edges are emitted in the
  canonical order — first-cap perimeter (`n` edges), last-cap perimeter (`n`),
  segment side seams (`n * s`), interior ring edges (`n * (s - 1)`) — for a
  total of `n * (2 * s + 1)`. No geometry, validation, `structural_hash`,
  triangle emission, face-label order, face-ID derivation, or
  `impl BRepProvider` semantics were touched; the only non-test change beyond
  the new impl is adding `BRepEdgeId`/`BRepEdgeProvider` to the `crate::topology`
  import.
- `crates/cad-core/src/topology/edge_resolve.rs`: Added
  `OperatorNode::Sweep(op) => Ok(op.brep_edge_ids(owner))` to the direct-provider
  match arms, so Sweep no longer falls through to the catch-all
  `BRepResolveError::TopologyChangingOperator`. Refreshed stale prose that
  classified Sweep as topology-changing for edge resolution: module doc, the
  `brep_edge_ids_for_node` doc bullets, the architectural-posture paragraph
  (`four` → `five` direct providers), and the catch-all comment. Boolean
  remains in the topology-changing category throughout.

### Tests

- `sweep.rs::sweep_brep_edge_ids_count_is_n_times_2s_plus_1`: pins the
  `n * (2 * s + 1)` edge count — square/2-point → 12, triangle/3-point → 15,
  square/4-point → 28.
- `sweep.rs::sweep_brep_edge_ids_follow_canonical_order_2_point_path`: builds
  the full expected `Vec<BRepEdgeId>` from `brep_face_ids` output and asserts
  byte-equality for the square/2-point case (first-cap, last-cap, segment side
  seams; no interior ring edges for a one-segment path).
- `sweep.rs::sweep_brep_edge_ids_follow_canonical_order_multi_segment_path`:
  same for the triangle/3-point case (15 edges) and additionally spot-pins one
  first-cap edge, one last-cap edge, one segment side seam from each of the two
  segments, and one interior ring edge at their canonical offsets.
- `sweep.rs::sweep_brep_edge_ids_stable_under_coordinate_change`: rebuilds with
  the same `profile_count`/`path_segment_count` but different profile and path
  coordinates and asserts byte-identical edge IDs for the same owner.
- `sweep.rs::sweep_brep_edge_ids_repeated_calls_byte_identical`: determinism of
  repeated calls.
- `edge_resolve.rs::resolver_direct_sweep_matches_direct_provider` (new):
  resolves `OperatorNode::Sweep` through `brep_edge_ids_for_node` and asserts
  `Ok` exactly equal to the direct `SweepOp::brep_edge_ids` call for both a
  2-point path (12 edges) and a multi-segment path (15 edges).
- `edge_resolve.rs::error_topology_changing_operator_carries_correct_kind`
  (updated): the Sweep block was removed; the test now pins Boolean only as
  the remaining v0 topology-changing operator. Doc comment refreshed.
- `brep_edge_chain_resolution_smoke.rs::sweep_resolves_directly_via_edge_provider`
  (replaces `sweep_returns_topology_changing_error`): asserts
