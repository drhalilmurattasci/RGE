## What I Changed

### Source

- `crates/editor-shell/src/render_path.rs`:
  - **`init_render_state` split into two phases.** Phase 1 runs
    unconditionally on production `resumed`: constructs the
    `winit::Window`, `GfxContext`, `SurfaceContext`, then (after the
    Phase 2 branch) constructs `EguiHost` + clones the
    `Arc<InspectorHandoff>`. Phase 2 is gated on `has_cad_scene ||
    has_prebuilt_mesh` and delegates to the unchanged
    `init_render_state_post_surface` helper (so the helper's mesh-side
    assertions remain intact and only run when meshes are actually
    available). In the empty-world branch the freshly-constructed
    `GfxContext` is stashed into `self.gfx_ctx` so the unconditional
    egui-host construction block (which reads `self.gfx_ctx.as_ref()`)
    succeeds. Updated the method's docstring to describe the two-phase
    contract.
  - **Phase 2 guard preserved.** The cuboid pipeline + render-mesh
    upload + frame-graph substrate + per-mesh material build still run
    only inside `init_render_state_post_surface`, which is now called
    only inside the `has_cad_scene || has_prebuilt_mesh` branch. The
    helper's contract is unchanged; the mesh-side debug_assert pair
    (`!materials.is_empty()`, `uploaded_meshes.len() == materials.len()`)
    is still reached exactly when a mesh source exists.
  - **`render_frame` early-dispatch.** A new guard at the top of
    `render_frame` checks `self.pipeline.is_none() ||
    self.compiled_frame_graph.is_none()`. Either being absent means
    Phase 2 did not run (both fields are stashed together at the end
    of `init_render_state_post_surface`, so partial state is not
    possible). When that branch fires, `render_frame` returns the
    result of `render_frame_egui_only` directly — the cuboid /
    frame-graph / depth-view code path is never entered.
  - **New private helper `render_frame_egui_only`.** Acquires the
    surface texture (with the same skip-on-transient policy as the
    cuboid path), issues a clear-only render pass to `DEFAULT_CLEAR`,
    publishes the inspector handoff via `self.inspector_snapshot()`,
    runs `EguiHost::render` against the same encoder, submits, presents,
    and returns `true`. Returns `false` when any of `surface_ctx` /
    `window` / `gfx_ctx` / `egui_host` is `None` so the
    "`EguiHost`-exists ⇒ true" contract from the acceptance criteria
    holds. Never references the cuboid pipeline, frame graph, depth
    view, or `encode_main_pass`.

- **`crates/editor-shell/src/lifecycle/mod.rs`** was NOT touched. The
  task allowed editing only if a real lifecycle tolerance hole existed
  for the Phase 1-only state. The audit traced through every
  lifecycle-side consumer of cuboid render fields:
  - `ApplicationHandler::resumed` propagates the `init_render_state`
    error and never inspects pipeline-side state directly.
  - `window_event::WindowEvent::Resized` calls `resize_render_path`,
    which already gates the camera UBO update on
    `self.gfx_camera.as_ref()` and the frame-graph rebuild on
    `self.compiled_frame_graph.is_some()`; both are `None` in Phase
    1-only state and the existing `if let` / `if` checks make those
    branches no-ops. The surface resize itself uses only `gfx_ctx` +
    `surface_ctx`, both of which are populated by Phase 1.
  - `window_event::WindowEvent::RedrawRequested` calls
    `self.tick_redraw()` (no render-state assumptions) then
    `self.render_frame()` (now dispatches to the egui-only path
    cleanly).
  - `window_event::WindowEvent::MouseInput` calls `handle_left_click`,
    which is guarded on `projection.is_some()`; world-only shells have
    `projection == None`, so face-pick stays a no-op — pre-existing
    contract, not a new hole.
  - `handle_asset_reload` short-circuits when `glb_source_path ==
    None`, which is the case for `--scene` launches.
  No new tolerance was required; the file was therefore left
  byte-identical, as the task scope explicitly permitted.
