## Goal

Implement GitHub issue #46 by replacing the `rge-anim-graph` stub with the
minimal Phase 8 graph-foundation reuse slice: a thin `AnimGraph` wrapper over
`rge_kernel_graph_foundation::Graph` that stores data-only animation states and
transition edges with deterministic IDs and graph-foundation error passthrough.
This establishes the animation graph substrate only; traversal, evaluation,
blend trees, runtime behavior, editor behavior, and renderer-tier integration
remain out of scope.

## Deliverables

- `crates/anim-graph/Cargo.toml` depends on
  `rge-kernel-graph-foundation = { workspace = true }` and adds no other
  dependency.
- `crates/anim-graph/src/lib.rs` exposes a minimal `AnimGraph` backed by
  `Graph<AnimState, AnimTransition>`.
- `AnimState` is a public data-only state payload with a deterministic string
  key, for example a single `key: String` field plus a small constructor.
- `AnimTransition` is a public data-only transition payload with a deterministic
  string key or trigger, for example a single `trigger: String` field plus a
  small constructor.
- `AnimGraph::new` and `Default` construct an empty graph.
- `AnimGraph::add_state` inserts an animation state, derives its `NodeId`
  deterministically from the state key bytes with `NodeId::from_bytes`, and
  returns the inserted `NodeId`.
- `AnimGraph::add_transition` or equivalently named connection method inserts a
  directed transition between existing state `NodeId`s, derives its `EdgeId`
  deterministically from `(src, dst, transition key/trigger)`, and returns the
  inserted `EdgeId`.
- Mutating methods preserve graph-foundation failures as passthrough errors,
  preferably through a small `AnimGraphError(pub GraphError)` newtype with
  `From<GraphError>`, `Display`, and `std::error::Error`.
- The wrapper exposes only minimal structural inspection needed by tests, such
  as `node_count` and `edge_count`; it does not expose traversal or evaluation.
- Focused inline unit tests in `src/lib.rs` cover stable `NodeId` derivation,
  duplicate state rejection, successful transition insertion, duplicate
  transition rejection, dangling endpoint rejection, and empty/default counts.
- A completed execution report packet for this dispatch.

## Acceptance Criteria

- `cargo test -p rge-anim-graph` exits 0.
- The crate no longer contains only the stub documentation; it has a real
  `AnimGraph` wrapper over `rge_kernel_graph_foundation::Graph`.
- Animation state and transition payloads remain data-only. They must not carry
  playback, blending, sampling, runtime scheduling, editor metadata, renderer,
  ECS, or asset-loading behavior.
- `NodeId` derivation is deterministic for equivalent state keys across fresh
  `AnimGraph` instances.
- `EdgeId` derivation is deterministic for equivalent `(src, dst, transition
  key/trigger)` tuples and rejects identical duplicate transitions through the
  underlying graph-foundation duplicate-edge path.
- Duplicate state insertion surfaces the underlying
  `GraphError::DuplicateNode` value without converting it into an opaque or
  string-only error.
- Transition insertion with an absent source or destination surfaces the
  underlying `GraphError::DanglingEndpoint` value without converting it into an
  opaque or string-only error.
- No traversal, topological sorting, cycle detection, evaluation, blend tree,
  state-machine runtime, editor, renderer-tier, asset, or ECS behavior is added.
- Tracked project-file changes are limited to
  `crates/anim-graph/src/lib.rs` and `crates/anim-graph/Cargo.toml`, plus the
  protocol-required execution report packet.

## Planner Notes

`crates/material-graph/src/lib.rs` is the closest read-only precedent: it wraps
`Graph`, derives IDs at the domain wrapper boundary, and keeps semantics
domain-local without adding evaluation or renderer behavior. Use that pattern
for scale and restraint, not as permission to touch material graph code.

The intended anim-graph surface is deliberately smaller than a real animation
state machine. Treat state keys and transition keys/triggers as opaque data
used solely for deterministic ID derivation and duplicate detection. Any
feature that would answer "what state is active?", "can this transition fire?",
"what pose results?", or "how should an editor draw this?" belongs in a later
dispatch.
