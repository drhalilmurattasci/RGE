## Goal

Add a same-phase retry for the two read-only model review phases in
`Invoke-AiDispatchLoop.ps1`: Claude plan-gate review and Codex control review.
Each eligible phase may retry once after an infrastructure or model-call
failure, so transient CLI/model failures do not abort an otherwise valid
dispatch. The retry must be local to the failed read-only phase and must not
change semantic verdict handling, correction loops, publish behavior, GitHub
API behavior, schemas, or the Codex stall watchdog's final failure wording.

## Deliverables

- Add a small, loop-local retry mechanism in `Invoke-AiDispatchLoop.ps1` that
  retries only these read-only phases after infrastructure or model-call
  failure:
  - `Invoke-ClaudePlanGate`, which calls Claude with `--permission-mode plan`
    and extracts `GATE_VERDICT`.
  - `Invoke-CodexControl`, which calls Codex with the `read-only` sandbox and
    writes the dispatch run-dir `codex.control.roundN.json` artifact.
- Each eligible phase may make at most one same-phase retry after the initial
  failed attempt. There must never be more than two total attempts for one
  phase invocation.
- A retry-eligible failure is a thrown or terminating failure from the model
  call path for that phase, including timeout, stalled Codex output,
  non-zero CLI exit, missing output, malformed/missing marker output, malformed
  control JSON, or model-reported CLI error. The retry should re-run the same
  phase with the same prompt inputs and round/revision number, not advance to
  another phase.
- Preserve the existing final failure messages after retry exhaustion. In
  particular, the final Codex stall failure must still use the current wording:
  `codex exec stalled: no log growth for ... after first output. Killed process
  tree. See ...`.
- Emit clear loop output for retry activity without changing any JSON schemas
  or JSONL trace schema:
  - when a same-phase retry is about to run,
  - when a retry succeeds, and
  - when the retry is exhausted and the original phase is failing.
- Keep retry output in normal loop stdout/stderr only. Do not add structured
  trace fields, sidecar fields, schema fields, persistent dashboards, or new
  log formats.
- Ensure retry attempts use distinct or safely overwritten phase artifacts so a
  failed first attempt cannot be mistaken for the successful retry result.
  Acceptable approaches include deleting/replacing the phase's previous output
  before retry, using attempt-suffixed transient logs and promoting the final
  result, or another compact implementation that leaves the existing public
  artifact names understandable.
- Preserve existing semantic verdict behavior:
  - Claude plan-gate `approve` finalizes the task and proceeds.
  - Claude plan-gate `needs_changes` consumes a plan revision and routes back
    to Codex plan-fill without same-phase retry.
  - Claude plan-gate `block` aborts without same-phase retry.
  - Codex control `pass` ends successfully without same-phase retry.
  - Codex control `needs_changes` writes a correction packet when correction
    rounds remain, or fails with the existing exhausted-round behavior when
    they do not, without same-phase retry.
  - Codex control `block` aborts without same-phase retry.
- Do not make any other phase retryable. In particular, do not add same-phase
  retry to Codex plan-fill, Claude execution, Claude readiness probe, Codex
  preflight audit, verification, correction packet generation, correction
  execution, queue publish, queue relabel, issue creation, branch operations,
  GitHub API calls, or any queue/auto runner behavior.

## Acceptance Criteria

- The only source file changed is `Invoke-AiDispatchLoop.ps1`.
- Claude plan-gate model-call failures are retried once in the same
  plan-gate revision before the dispatch aborts.
- Codex control model-call failures are retried once in the same control round
  before the dispatch aborts.
- A successful retry leaves normal downstream state intact: the approved
  Claude gate can finalize the TASK, and a successful Codex control retry can
  produce the same final control JSON/readiness output expected by the queue.
- A retry success prints a human-readable loop message that identifies the
  phase and attempt outcome.
- Retry exhaustion prints a human-readable loop message that identifies the
  phase and exhaustion, then preserves the current final `Fail` message from
  the underlying failure path.
- Codex stall watchdog behavior is unchanged: the watchdog still kills a
  stalled Codex process using the existing no-log-growth logic, still returns
  the same stalled/timed-out result internally, and after retry exhaustion the
  final stderr wording remains the existing `codex exec stalled: no log growth
  for ... after first output. Killed process tree. See ...` message.
- `ModelTimeoutSec`, `VerifyTimeoutSec`, `CodexStallThresholdSec`, and
  `MaxPlanRevisions` / `MaxCorrectionRounds` semantics are unchanged except
  for the single same-phase retry around the two eligible read-only phases.
- Claude `needs_changes`, Claude `approve`, Codex `pass`, and Codex
  `needs_changes` are treated as semantic verdicts, not retryable failures.
- Claude `block`, Codex `block`, blocked/failed Claude execution,
  verification failure, verification timeout, exhausted plan revisions,
  exhausted correction rounds, and task/correction packet validation failures
  remain non-retryable unless they arise specifically as the eligible model
  call failure being retried in the read-only phase.
- No retry is added to Codex plan-fill, Codex correction generation, Claude
  execution, Claude readiness probe, Codex preflight audit, verification, queue
  publish, queue relabel, issue creation, branch operations, GitHub API calls,
  or any script other than `Invoke-AiDispatchLoop.ps1`.
- No `.ai/codex_control.schema.json`, handoff schema, JSONL trace schema,
  packet template, queue log format, or GitHub label behavior changes.

## Constraints / Non-Goals

- This dispatch is only for same-phase retry of Claude plan-gate and Codex
  control read-only model review phases.
- Do not alter the meaning of model verdicts. A semantic verdict is a
  completed phase result, even when it blocks progress or requests changes.
- Do not add retry policy knobs, environment variables, command-line
  parameters, dashboards, health summaries, issue labels, docs, schemas,
  tests, dependencies, or helper scripts.
- Do not change or wrap the queue runner's existing retry policy. This is an
  inner-loop same-phase retry, not an issue requeue or publish retry.
- Do not change commit, push, publish, branch, label, issue selection,
  autonomous scheduling, or GitHub API behavior.
- Do not change the canonical verification gate or make verification failures
  eligible for same-phase retry.
- Do not change preflight audit behavior. If `Invoke-CodexPreflightAudit`
  fails, it must still fail without same-phase retry.
- Avoid broad exception handling that hides failures after the single retry.
  After the retry is exhausted, the dispatch must fail loudly through the same
  failure path and wording used today.

## Planner Notes

The target call sites are `Invoke-ClaudePlanGate` and `Invoke-CodexControl`.
Both are read-only review phases from the dispatch design perspective: the
Claude gate uses plan permission mode and must not edit files, while Codex
control uses the `read-only` sandbox and writes only its run-dir review
artifacts. A compact helper that invokes a scriptblock, prints retry messages,
and rethrows the final caught exception after one retry is likely sufficient.

Be careful with the current `Fail` helper: it writes to stderr and exits the
PowerShell process. If a retry wrapper cannot catch a `Fail` from the current
call stack, refactor only the eligible phase call path enough to let those two
read-only phases signal retryable model-call failures without changing the
public final failure text. Do not turn unrelated `Fail` calls into retryable
exceptions.
