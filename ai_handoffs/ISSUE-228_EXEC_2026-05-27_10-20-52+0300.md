  re-armed once `ai-auto` was stripped." paragraph): appended a
  "**Manual salvage protocol.**" block covering:
  - **Salvage branch.** Preserve useful work on a clearly named
    salvage branch such as `ai-dispatch/ISSUE-<n>.salvage`, branched
    from the original `ai-dispatch/ISSUE-<n>` branch; the original
    dispatch branch is preserved (cross-references §18.5).
  - **Audit log.** If the queue audit log under `ai_dispatch_logs/`
    is missing or incomplete, keep or add a branch-local
    `ai_dispatch_logs/log_*.md` recording the original dispatch id,
    salvage reason, commands run, verify result, and any control
    verdict available before salvage.
  - **No direct merge.** Do not fast-forward or merge the salvage
    branch onto `main` until a human reviews it; the auto-publish
    gate does not run for salvage branches.
  - **Issue comment.** Comment on the issue with the salvage
    branch, commit SHA(s), verify result, Codex control verdict (or
    a note none was available), and manual recovery reason in plain
    language.
  - **Label cleanup.** Strip misleading queue labels —
    `ai-dispatch-running`, `ai-dispatch-retry`, stale
    `ai-dispatch-failure-*` taxonomy labels, and `ai-dispatch-failed`
    when it is being superseded by positive salvage; keep
    `ai-dispatch-failed` for abandoned salvage. Keep
    `ai-dispatch-done` only for positive salvage; strip it
    otherwise.
  - **Issue close policy.** Close as completed only after the
    useful work is merged or has been explicitly accepted;
    otherwise leave open or close as not planned with a one-line
    reason.
  The block closes with a paragraph reiterating that this protocol
  complements §14.8's `ai-auto` selector re-arming requirement and
  §14.13's terminal label reconciliation helper, separating the
  queue-owned terminal label states from the human-owned transition
  out of them.
- `AI_DISPATCH_AUTOMATION.md` (§14.13, before the final `---` that
  precedes §15): appended a "**Terminal failure-taxonomy labels.**"
  block that names each of the seven existing taxonomy labels with
  a one-line purpose:
  - `ai-dispatch-failure-stall` — Codex watchdog stall (distinct
    from generic timeout).
  - `ai-dispatch-failure-timeout` — wall-clock timeout without a
    watchdog stall signature.
  - `ai-dispatch-failure-blocked` — `EXEC_STATUS: blocked` (designed
    halt; cross-references §14.10).
  - `ai-dispatch-failure-verification` — `.ai/dispatch.verify.ps1`
    or CORRECTION-retry exhaustion failed.
  - `ai-dispatch-failure-control` — control `block` or exhausted
    `MaxCorrectionRounds` on `needs_changes`.
  - `ai-dispatch-failure-publish` — loop succeeded but the queue's
    commit/push failed (fast-forward refused, push auth/network).
  - `ai-dispatch-failure-unknown` — terminal failure that did not
    match any classification above.
  The block explicitly states these labels are "descriptive triage
  signals layered on top of `ai-dispatch-failed`", "not retry
  selectors by themselves", and "participate in neither issue
  selection, cap counting, halt checks, retry eligibility,
  publishing, branch archival, nor cleanup"; it cross-references
  task #52 in `.ai/dispatch.tasks.md` for the queue behavior that
  applies them, the surrounding §14.13 paragraph for the
  reconciliation helper, and the §14.8 manual salvage block for
  stale-label stripping during human recovery.
