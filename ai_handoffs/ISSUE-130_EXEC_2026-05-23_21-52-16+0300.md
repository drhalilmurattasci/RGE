## 5-Question Cap-v2 / Label-Aging Answer Block

### Q1. Lifetime vs rolling-window count semantics?

**Current implemented semantics: lifetime, all-states, no age filter.**

`Invoke-AiDispatchAuto.ps1` builds the cap-driving count via this `gh`
call (verbatim from lines 364–366, quoted in Gate V3):

```powershell
    $allAuto = Get-IssuesJson @(
        'issue', 'list', '--repo', $repoSlug, '--label', $autoLabel,
        '--state', 'all', '--limit', '200', '--json', 'number')
```

The cap comparison is `$allAuto.Count -ge $MaxAutonomousTasks` (line 367).
The query has three properties that together fix the semantics:

1. **`--state all`** — counts open AND closed issues. A closed
   `ai-auto`-labelled issue stays in the count.
2. **No `--created` / `--updated` filter** — there is no temporal
   filter at all. Age does not affect the count.
3. **`--label ai-auto`** — the only inclusion criterion is the
   `ai-auto` label. Removing the label from an issue (the only
   mechanically-enforceable salvage step per §14.8 of
   `AI_DISPATCH_AUTOMATION.md`, quoted in Gate V5) is the ONLY way an
   issue exits the count today.

GitHub evidence confirms this:

- `gh issue list --repo RustCADs/RGE --label ai-auto --state all
  --limit 200` returns 101 issues (Gate V7).
- The REST cross-check (Gate V8) agrees: 101 issues, 1 open, 100 closed.
- 100 of the 101 issues are CLOSED, which under any rolling-window
  policy with `closedAt` filtering would be a candidate for removal —
  yet they all currently count. This is the lifetime semantics in
  action.
- The cap has been hit once already at exactly this boundary (audit
  ISSUE-128, PR #129, 2026-05-23). The widening of
  `[ValidateRange(1, 100)]` → `[ValidateRange(1, 200)]` in commit
  `e7104c0` was a runtime accommodation for the +1 needed for this
  dispatch, not a doctrinal shift away from lifetime counting.

**Summary**: Lifetime/all-states is the current code reality. The
choice between lifetime and rolling-window is a doctrinal fork that the
code has not yet made; the cap-stop ISSUE-128 was the first event that
forced the fork into the open.

### Q2. If rolling, what age threshold (30 / 60 / 90 days)?

**Empirical observation: with today's `ai-auto` population, 30 / 60 / 90
days are observationally indistinguishable. All three buckets would
preserve the entire 101-issue population because the oldest `ai-auto`
issue is only ~6.01 days old.**

From Gate V7, the createdAt age distribution today (2026-05-23) is:

| Threshold | Issues younger than threshold | Issues older (= candidates to age out) |
|---|---:|---:|
| 30 days | 101 | 0 |
| 60 days | 101 | 0 |
| 90 days | 101 | 0 |

Per-day createdAt counts confirm the population is compressed into a
6-day window: 7 (2026-05-17) → 23 → 24 → 26 → 0 → 6 → 15 (2026-05-23).
The min/median/max createdAt age in days is 0.00 / 3.87 / 6.01; the
min/median/max closedAt age (for the 100 closed issues) is
0.28 / 3.94 / 5.94.

**Tradeoff analysis the read-only evidence supports:**

- **30 days**: tightest window. With current 6-day age spread, no
  candidate would age out today, so the 30-day window is currently
  equivalent to lifetime. It becomes meaningful only after the
  population spans more than 30 days — which today's data cannot
  yet validate.
- **60 days**: middle window. Same observation as 30 days for today
  but with twice the slack; would absorb roughly two months of
  ~20-dispatch-per-day activity at the recently observed rate.
- **90 days**: most permissive window. Would only age out an issue
  after a full quarter. Most generous to "ongoing relevance" of an
  `ai-auto` entry; mostly identical to a lifetime cap from the
  perspective of a project under three months old. The current
  population is fully under 7 days, so a 90-day window today is
  indistinguishable from lifetime.

**Direction the evidence points (recommendation, not implementation):**

The read-only evidence does not distinguish 30 / 60 / 90 today, so any
choice has to be justified on grounds other than current population
distribution. The two non-trivial pieces of evidence the audit can
add are:

1. **Recent dispatch rate** — the ai-auto population grew from 0 to
   ~100 in ~6 days, i.e. ≈ 17 issues per day during an active arc.
   At that rate a 30-day window allows ≈ 510 in-window
   issues before the cap re-engages; 60 days ≈ 1020; 90 days ≈ 1530.
   None of these is meaningful unless `-MaxAutonomousTasks` itself is
   also re-tuned, which is currently 100 (the prior persisted
   value) or 200 (the new ValidateRange ceiling).
2. **Doctrinal anchor** — §17.4 of `AI_DISPATCH_AUTOMATION.md` (Gate
   V5) commits to "One task per dispatch, not a batch" and to
   `-MaxAutonomousTasks` being raised "one at a time, not in bulk."
   A short window (30 days) keeps the cap behavior closer to "every
   batch reviewed" because aged-out issues re-enter selectability
   sooner; a long window (90 days) drifts toward "lifetime cap with
   slow leak."

A 60-day rolling window on `closedAt` is the smallest defensible
default the read-only evidence supports: it preserves the §17.4
human-gated posture (each new batch is reviewed within two months
of the last) while still giving the cap a path back to zero without
manual scrubbing.

**Caveat**: today's data cannot empirically rule out any of the three;
the choice is a forward-looking policy call. This is not a halt
condition because the evidence is sufficient to recommend 60 days
**conditionally on** the doctrinal anchor in §17.4; a different
anchor (e.g. "monthly cleanup") would favor 30 days.

### Q3. Cleanup mechanism: manual command, script mode, or doctrine only?

Three mechanisms can implement an `ai-auto` rolling cleanup. The
audit compares them (does not implement any).

**Option A — manual `gh` command per-batch** (smallest blast radius):

The §14.8 salvage recipe already documents the exact `gh` command:

```powershell
gh issue edit <num> `
  --remove-label ai-auto `
  --remove-label ai-dispatch-failed `
  --remove-label ai-dispatch-retry
```

Extending it to an aging script for the cap circuit means the human
runs (or copies from doctrine) a one-liner like:

```powershell
# Read-only candidate listing, no mutation:
gh issue list --repo RustCADs/RGE --label ai-auto --label ai-dispatch-done `
  --state closed --search 'closed:<2026-04-23' --json number,closedAt
```

…and then runs `gh issue edit --remove-label ai-auto` per candidate.

- **Pros**: zero script change; zero scheduler change; every removal
  is a deliberate human act; the audit trail is the issue history
  itself; aligns with §17.4 "human-owned publish/merge" and "raised
  one at a time."
- **Cons**: requires the human to remember (or check doctrine);
  manual on each cleanup; no central place that records the policy.

**Option B — bounded script mode on `Invoke-AiDispatchAuto.ps1`**
(medium blast radius):

Add a `-Mode AgeAuditReport` (read-only) or `-Mode AgeOutAiAuto`
(mutating) switch. Two sub-shapes:

- `-Mode AgeAuditReport`: read-only diagnostic. Prints which issues
  are candidates at 30 / 60 / 90 days and the resulting cap count
  after each. No mutation. Safest mid-tier.
- `-Mode AgeOutAiAuto`: mutating. Removes `ai-auto` from candidates.
  Requires a separate human-approval prompt or a `-Confirm` flag.

- **Pros**: codified threshold; reproducible run; can be
  cron-scheduled or chain off the existing `Invoke-AiDispatchAuto.ps1`
  scheduled task.
- **Cons**: doubles the surface area of the driver; the driver's
  scope expands from "select + run" to "select + run + age";
  mutating sub-mode contradicts §17.4's "human-owned" stance unless
  manually invoked.

**Option C — doctrine-only guidance** (smallest mechanical change;
keeps existing semantics):

Add a new §17.5 (or §14.12) to `AI_DISPATCH_AUTOMATION.md` titled
something like "Cap aging — when and how to scrub `ai-auto`". The
section would:

- Affirm the current code semantics (lifetime, all-states, label-keyed).
- State the chosen threshold (e.g., "remove `ai-auto` from any
  closed `ai-dispatch-done` issue whose closedAt is older than 60
  days").
- Cite §14.8's existing `gh issue edit --remove-label ai-auto`
  recipe and reuse it (no new mechanism).
- List the labels that MUST remain (see Q4).
- Pin a re-audit cadence (e.g., "re-audit whenever the active
  ai-auto count is within 10 of `-MaxAutonomousTasks`").

- **Pros**: no script change, no scheduler change, no new code path.
  Reuses §14.8's existing mechanical lever. Easiest to reverse.
  Aligns with the precedent of §14.8 itself being a doctrine entry
  with a tiny `gh` recipe.
- **Cons**: relies on humans following doctrine; no enforcement.

**Audit recommendation (not implementation)**: Option C is the
smallest safe step. Option A is what humans already do today
(implicitly, via §14.8's salvage recipe). Option B prematurely
expands the driver's responsibility. Option C codifies the policy
choice (the threshold, the keep-list) without committing to new
code or new scheduling — which is exactly what an
under-3-months-old policy needs while data accumulates to validate
the threshold.

### Q4. What labels should remain on aged-out issues?

**Current doctrine** (§14.8 + §17.2, quoted in Gate V5): when an
autonomous dispatch is *salvaged* (a manual, conscious act
distinguishing from "aged out"), the human MUST scrub
`ai-auto` + `ai-dispatch-failed` + `ai-dispatch-retry`. The doctrine
explicitly says "Keep `ai-dispatch-done` if the dispatch's substantive
deliverable landed (positive salvage); strip it if the dispatch was
abandoned." `ai-dispatch` itself is the queue label and is
mechanically removed by `Invoke-AiDispatchQueue.ps1` during the
transition through `ai-dispatch-running` to `ai-dispatch-done` /
`ai-dispatch-failed`, so it is rarely present on a closed issue
that's a candidate for aging.

**Proposed future aging policy** (recommendation only; not
implemented):

| Label | Action on aged-out cleanup | Rationale |
|---|---|---|
| `ai-auto` | **Remove.** This is the cap-circuit lever; removing it is the entire point of aging. | Mechanically equivalent to §14.8's salvage step. After removal the issue stops being counted by `$allAuto.Count` (Gate V3, lines 364–366). |
| `ai-dispatch-done` | **Keep.** Preserves the audit trail of which auto-dispatches landed. | §14.8's "Keep `ai-dispatch-done` … positive salvage" treats this tag as the durable audit-trail marker. |
| `ai-dispatch-failed` | **Keep on aged-out issues**, even though §14.8 says to remove it during *salvage*. | Salvage is a conscious "re-arm this task entry" act; aging is "this entry is no longer relevant to the cap." A failed-and-then-aged issue still records the failure for retrospective review. **This is the key distinction between salvage and aging policy.** |
| `ai-dispatch-retry` | **Keep on aged-out issues**, for the same reason as `ai-dispatch-failed`. | Same audit-trail justification. Aged-out cleanup is not a re-arming act. |
| `ai-dispatch` | Already removed by the queue runner before closure in normal flow; if present on an aged-out issue, **remove** (matches queue-runner state-machine semantics). | The label means "queued"; an aged closed issue is by definition not queued. |

**Crucial distinction from current doctrine**: §14.8's salvage rule is
designed to *re-arm a task entry* against the selector. Aging is
*not* a re-arming act — the task entry has already been served by the
original dispatch (or failed and the failure has been observed). So
the aged-out cleanup should be narrower than the salvage cleanup:
remove only `ai-auto`, and keep `ai-dispatch-failed` /
`ai-dispatch-retry` as durable failure metadata.

This distinction is what makes Q4 a genuine policy question rather
than a code question. The current doctrine in §14.8 does not yet
address aging because the cap had not yet been hit at the time it
was written; ISSUE-128 was the first event that made the
distinction operationally meaningful.
