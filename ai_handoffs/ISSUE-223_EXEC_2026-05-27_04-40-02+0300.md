- `.github/workflows/powershell.yml`: new dedicated Windows PowerShell CI
  workflow. Runs on `windows-latest`. Three steps under `shell:
  powershell` (Windows PowerShell 5.1, matching the local
  `#Requires -Version 5.1` scripts and the verification commands in the
  TASK packet):
    1. Install Pester 5+ and PSScriptAnalyzer from PSGallery into
       `CurrentUser` scope.
    2. `Invoke-Pester -Path 'tools/dispatch-tests' -CI` — `-CI` sets
       `Run.Exit = $true` so a failing `It` non-zero-exits the step.
    3. PSScriptAnalyzer over tracked `*.ps1` / `*.psm1` enumerated by
       `git ls-files`, explicitly excluding `(^|/)\.ai/dispatch-` scratch.
       Iterates over files (PSScriptAnalyzer `-Path` is `<String>`, not
       `<String[]>`), applies the committed settings file, fails the
       step on any Error or Warning finding.

  Triggered on `push` to main, on PRs into main, and via manual
  `workflow_dispatch`. Permissions reduced to `contents: read`.

## Open Questions for Reviewer

1. The eight-rule exclusion list in `PSScriptAnalyzerSettings.psd1` is
   the load-bearing piece of the analyzer gate. Each exclusion is
   justified inline, but the list is broader than I would prefer; a
   follow-up issue could narrow it by either (a) renaming the
   unapproved-verb helpers under a coordinated cross-script refactor,
   (b) replacing `Write-Host` UI lines with a dedicated streaming
   channel, or (c) adding `[CmdletBinding(SupportsShouldProcess)]` to
   the snapshot-mutation helpers. Out of scope here, but worth a
   `Status.md` `Watchlist` entry.

2. The Pester test passes `$QueueLabel = 'pester-queue-label'` and
   `$runLabel = 'pester-queue-label-running'` via `Set-Variable -Scope
   Script` to mirror what the production top-level body normally
   computes. The matching production strings (`ai-dispatch`,
   `ai-dispatch-running`) would also work; I chose distinct synthetic
   values so the test can assert that the writer actually consumes the
   in-scope variables rather than hard-coded literals.

3. The Pester regex assertion is `[regex]'\$[A-Za-z_][A-Za-z0-9_]*'`
   per the TASK packet. PowerShell also accepts the unusual identifier
   form `${name with spaces}`; the TASK regex (and therefore the test)
   intentionally does not flag that variant, because the writer's
   here-string never uses it. If the writer is ever changed to emit
   `${...}` tokens, the regex would need a second alternative.
