## Goal

Add a label-only terminal failure taxonomy to `Invoke-AiDispatchQueue.ps1` for
GitHub issue #190. The queue already halts autonomous progress by marking
terminal failures with `ai-dispatch-failed`; this dispatch must preserve that
halt behavior while adding durable, more specific failure-class labels for
human triage and later policy analysis. The change must not alter retry
eligibility, recovery routes, issue selection, cap counting, publishing, branch
policy, or any dispatch loop behavior.

## Deliverables

- Add idempotent creation for these taxonomy labels in the existing queue
  label setup path that already creates `ai-dispatch`, `ai-dispatch-running`,
  `ai-dispatch-done`, `ai-dispatch-failed`, and `ai-dispatch-retry`:
  - `ai-dispatch-failure-stall`
  - `ai-dispatch-failure-timeout`
  - `ai-dispatch-failure-blocked`
  - `ai-dispatch-failure-verification`
  - `ai-dispatch-failure-control`
  - `ai-dispatch-failure-publish`
  - `ai-dispatch-failure-unknown`
- Add a compact queue-local helper that classifies terminal failed runs using
  only data already available after the dispatch loop and publish decision:
  `$loopExit`, `$loopText`, `$verdict`, `$execStatus`, `$publishFailed`,
  `$publishHardFailed`, and nearby queue-local state already computed in
  `Invoke-AiDispatchQueue.ps1`.
- The helper must produce at least one selected taxonomy label for every
  terminal failed run. A single primary label is sufficient and preferred.
  If no more specific class matches, select
  `ai-dispatch-failure-unknown`.
- Classification order must preserve the intended signal:
  1. Publish failure: select `ai-dispatch-failure-publish` for publish-pipeline
     failures.
  2. Blocked execution: select `ai-dispatch-failure-blocked` when
     `$execStatus -eq 'blocked'`.
  3. Codex watchdog stall: select `ai-dispatch-failure-stall` when loop text
     contains Codex watchdog stall wording, including `codex exec stalled` or
     `no log growth`.
  4. Generic timeout: select `ai-dispatch-failure-timeout` when loop text
     contains timeout wording and the stall class did not match.
  5. Verification failure: select `ai-dispatch-failure-verification` when loop
     text contains verification-gate failure wording.
  6. Codex control failure: select `ai-dispatch-failure-control` when loop
     text contains control-block or exhausted-control-change wording.
  7. Unknown: select `ai-dispatch-failure-unknown` otherwise.
- Apply taxonomy labels only when `$runFailed` is true and `$willRetry` is
  false. This includes failed retry attempts, blocked executions, and terminal
  publish failures, but excludes successful runs and first-attempt failures
  that are being re-queued for automatic retry.
- Preserve the existing `ai-dispatch-failed` label behavior unchanged:
  terminal failed issues must still receive `ai-dispatch-failed`; retry-queued
  failures must still avoid `ai-dispatch-failed`; successful issues must still
  avoid `ai-dispatch-failed`.
- Add the selected taxonomy label or labels to the existing terminal failure
  relabel command without adding any taxonomy label to the retry relabel path
  or the success relabel path.
- Extend label finalization verification so terminal failed issues require the
  selected taxonomy label or labels in addition to the existing expected
  terminal labels. A terminal failed issue that lacks its selected taxonomy
  label must make label finalization fail non-zero just like a missing
  `ai-dispatch-failed` label does today.
- Do not make taxonomy labels participate in issue selection, cap counting,
  halt checks, retry eligibility, branch archival, orphan recovery, publish
  eligibility, branch deletion, cleanup, or autonomous task selection.
- It is acceptable, but not required, to mention the selected taxonomy label or
  labels in the result comment for human readability. The durable output is the
  GitHub label set.

## Acceptance Criteria

- The only source file changed is `Invoke-AiDispatchQueue.ps1`.
- The existing queue labels are still created and used with their existing
  names: `ai-dispatch`, `ai-dispatch-running`, `ai-dispatch-done`,
  `ai-dispatch-failed`, and `ai-dispatch-retry`.
- The seven new taxonomy labels are created idempotently through the same
  `gh label create ... --force` setup style as the existing queue labels.
- Successful runs retain the existing successful label outcome: the issue has
  `ai-dispatch-done`, does not have `ai-dispatch-running`, does not have
  `ai-dispatch`, does not have `ai-dispatch-retry`, does not have
  `ai-dispatch-failed`, and receives no `ai-dispatch-failure-*` taxonomy label.
- Successful control-passed publish behavior is unchanged: publish eligibility,
  fast-forward merge, push, branch deletion, issue close behavior, and
  successful result comments are not changed except for unavoidable surrounding
  code movement if needed for the taxonomy helper.
- First-attempt retry-eligible failures retain the existing retry outcome:
  `$willRetry` stays true under the same conditions as before, the issue keeps
  `ai-dispatch`, receives `ai-dispatch-retry`, does not receive
  `ai-dispatch-failed`, and receives no `ai-dispatch-failure-*` taxonomy label.
- Terminal failed runs are exactly the cases where `$runFailed` is true and
  `$willRetry` is false. Those issues receive `ai-dispatch-done`,
  `ai-dispatch-failed`, and at least one selected `ai-dispatch-failure-*`
  taxonomy label, while losing `ai-dispatch-running`, `ai-dispatch`, and
  `ai-dispatch-retry`.
- Codex watchdog stalls are classified as `ai-dispatch-failure-stall`, not as
  `ai-dispatch-failure-timeout`, when loop text contains `codex exec stalled`
  or `no log growth`.
- Generic timeouts that are not Codex watchdog stalls are classified as
  `ai-dispatch-failure-timeout`.
- Blocked executions with `$execStatus -eq 'blocked'` are classified as
  `ai-dispatch-failure-blocked` and remain terminal, not retryable.
- Verification-gate failures and exhausted verification correction rounds are
  classified as `ai-dispatch-failure-verification` when their loop text carries
  the existing verification failure wording.
- Codex control blocks and exhausted control correction rounds are classified
  as `ai-dispatch-failure-control` when their loop text carries the existing
  control failure wording.
- Publish-pipeline terminal failures are classified as
  `ai-dispatch-failure-publish`.
- Any terminal failed run that does not match a more specific class receives
  `ai-dispatch-failure-unknown`.
- Label finalization verification checks for taxonomy labels only on terminal
  failed issues. It does not require taxonomy labels for successful issues or
  retry-queued issues.
- No retry eligibility, same-phase retry behavior, recovery route, queue
  selection filter, cap-counting behavior, publish behavior, branch policy,
  Auto Loop behavior, watchdog behavior, preflight audit behavior, JSONL trace
  schema/path behavior, Rust/Cargo behavior, docs, dashboards, workflows, or
  schemas change.

## Planner Notes

Use the post-loop/post-publish finalization block in `Invoke-AiDispatchQueue.ps1`
as the integration point. The existing variables `$runFailed`, `$runBlocked`,
and `$willRetry` already define whether a run is successful, retry-queued, or
terminal. Compute taxonomy labels after those values are known, and add them
only to the non-retry terminal failure relabel and verification path.

Prefer a small helper with case-insensitive text matching over new state,
schema, trace output, or issue queries. The classification helper should not
alter the loop output, read or write extra files, call GitHub, or affect
control flow. It should return `ai-dispatch-failure-unknown` if no specific
rule matches.
