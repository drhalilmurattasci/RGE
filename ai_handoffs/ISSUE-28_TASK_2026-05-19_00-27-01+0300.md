## Goal

Add one focused cad-core integration smoke for GitHub issue #28 proving that a
committed `CuboidOp -> FilletOp` graph chain resolves B-Rep identity at the
Fillet node through the public graph resolvers. The smoke must pin the current
chamfer-Fillet contract: face IDs inherit the upstream Cuboid face IDs across
radius and selected-edge changes, while edge IDs inherit only the surviving
upstream Cuboid edges and filter out the selected filleted edges.

## Deliverables

- A new cargo-discovered integration test file at
  `crates/cad-core/tests/fillet_brep_chain_resolution_smoke.rs`.
- A small public-API fixture that:
  - creates a deterministic non-cubic `CuboidOp`;
  - gets upstream face IDs via `BRepProvider::brep_face_ids`;
  - gets upstream edge IDs via `BRepEdgeProvider::brep_edge_ids`;
  - constructs `FilletOp::new(&cuboid, owner, selected_edges, radius)`;
  - builds and commits a `CadGraph` with `OperatorNode::Cuboid(cuboid)` feeding
    `OperatorNode::Fillet(fillet)`;
  - resolves at the Fillet node with `brep_face_ids_for_node` and
    `brep_edge_ids_for_node`.
- Assertions that every tested Fillet radius and selected-edge variation
  returns face IDs exactly equal to the direct upstream Cuboid face IDs.
- Assertions that every tested Fillet radius and selected-edge variation
  returns edge IDs exactly equal to the upstream Cuboid edge vector with the
  selected filleted edges removed, preserving order and byte identity for the
  surviving upstream edges.

## Acceptance Criteria

- `cargo test -p rge-cad-core --test fillet_brep_chain_resolution_smoke -- --nocapture`
  exits 0.
- The new smoke compiles as an integration test without adding or changing any
  Cargo manifest entry.
- The test uses existing public `rge_cad_core` APIs only. It may use the public
  graph scaffolding needed for an integration test (`CadGraph`,
  `OperatorNode`, `BRepOwnerId`, and ID types), but it must not import private
  modules or Fillet internals.
- The upstream reference vectors are produced directly from the Cuboid:
  `cuboid.brep_face_ids(owner)` mapped to `Vec<BRepFaceId>` in emission order,
  and `cuboid.brep_edge_ids(owner)` as `Vec<BRepEdgeId>` in emission order.
- The resolver assertions compare exact vectors with `assert_eq!`; the test
  must not sort, deduplicate, or otherwise normalize the face or edge vectors.
- Radius variation is covered with at least three valid positive radii for the
  same selected edge set. For every radius, resolved face IDs must equal all
  upstream Cuboid face IDs and resolved edge IDs must equal the upstream Cuboid
  edges with the selected filleted edges filtered out.
- Selected-edge variation is covered with at least three valid selections,
  including a single-edge selection, a multi-edge selection, and all 12 Cuboid
  edges. For every selection, resolved face IDs must equal all upstream Cuboid
  face IDs and resolved edge IDs must equal the upstream Cuboid edges with that
  exact selection filtered out.
- The all-12-edge selection must assert that the resolved Fillet edge vector is
  empty while the resolved Fillet face vector still contains the 6 upstream
  Cuboid face IDs.
- No direct `impl BRepProvider for FilletOp` or `impl BRepEdgeProvider for
  FilletOp` is added.
- No production source, Cargo metadata, existing tests, docs, scripts, schemas,
  or repository-governance files are changed.

## Planner Notes

`crates/cad-core/tests/fillet_smoke.rs` already has a compact resolver check for
Fillet face inheritance plus edge filtering. This dispatch adds the missing
cargo-discovered integration smoke named by issue #28, mirroring the public API
shape of `round_fillet_brep_chain_resolution_smoke.rs` while asserting the
opposite edge behavior: chamfer `FilletOp` filters selected edges, whereas
`RoundFilletOp` preserves them. Keep the new file small and focused; a helper
that builds one `CuboidOp -> FilletOp` chain per variation is enough.
