## Acceptance Criteria

- Current-state wording, where present, reflects these facts:
  - `Invoke-AiDispatchLoop.ps1` is the inner one-dispatch orchestrator and still
    does not commit or push.
  - `Invoke-AiDispatchQueue.ps1` is the outer GitHub issue queue runner.
  - The queue selects open issues labelled `ai-dispatch`, runs one issue at a
    time on `ai-dispatch/ISSUE-N`, and writes an `ai_dispatch_logs/log_*.md`
    audit file.
  - The queue commits the branch result and, unless `-NoPublish` is used,
    fast-forwards `main` and pushes `origin/main` only when the loop exits 0 and
    the latest Codex control verdict is `pass`.
  - Failed, blocked, non-pass, or publish-failed runs remain local for
    inspection and are labelled failed.
  - `-NoPublish` is the one-off switch for the old local-branch-only behavior.
  - Recurring `/loop` use is semi-attended and bound to the local authenticated
    CLI/session, not a headless cloud daemon.
  - Orchestrator dispatches normally emit TASK and EXEC packets only, plus a
    CORRECT packet only if a correction round runs; missing REVIEW or CLOSEOUT
    packets are not evidence that an orchestrator dispatch is unfinished.
- Wording about "human authorization" or "no automatic push" is either removed
  or narrowed so it applies only to the inner loop, manual protocol, fan-out
  worktrees, or `-NoPublish` runs, not to successful queue auto-publish runs.
- Historical snapshots and existing packet history are not broadly rewritten.
  Prefer current-state notes near the top of docs over mutating chronology.
- `Invoke-AiDispatchQueue.ps1` changes, if any, are comment-only and do not
  alter runtime behavior.
- Final `git status --short --branch` shows only allowed docs/comment edits plus
  the ISSUE-3 EXEC packet and optional generated sidecar.
