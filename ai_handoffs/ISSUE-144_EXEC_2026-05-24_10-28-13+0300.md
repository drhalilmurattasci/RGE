## Per-File Summary

`Wait-GitHubActions.ps1` (single hunk in the `while ($true)` loop):

- `--json` field list now: `databaseId,name,workflowName,status,conclusion,headSha,createdAt,url`.
- `$sortedMatchingRuns = @($matchingRuns | Sort-Object @{ Expression = { [datetime]$_.createdAt }; Descending = $true })` runs once per poll; `$latestByName` is then populated from `$sortedMatchingRuns` rather than re-sorting inline. This preserves the "latest run wins" rule used for default `namesToCheck` discovery.
- Per expected name (whether explicit `-WorkflowName` values or default-discovered workflow names), the resolved run is `$sortedMatchingRuns | Where-Object { $_.workflowName -eq $name -or $_.name -eq $name } | Select-Object -First 1`. Because `$sortedMatchingRuns` is already createdAt-descending, `-First 1` is the newest run matching either field — preserving "latest run per expected workflow" while extending matching to the workflow-identity field.
- No fuzzy, substring, or explicitly case-folded matching introduced. `-eq` and hashtable lookups retain PowerShell's default string comparison semantics, matching the pre-change behavior on the `name` field.
- Display row `Name` column still shows `$run.name` when a run is found, mirroring the prior output; `missing` rows fall back to the requested expected name.
- Branch filtering (`gh run list --branch $Branch`), commit filtering (`$_.headSha -eq $Commit`), allowed-success set (`success`, `skipped`, `neutral`), deadline check before each poll and before each sleep, sleep cap to remaining time, and exit codes (0 / 1 / 2) are untouched.
