## Acceptance Criteria

- `KeybindingTarget` identifies a menu entry by both extension point id and entry
  id, using the existing `ExtensionPoint` / `EntryId` concepts rather than a
  shell, host, settings, or command-bus identifier.
- `KeybindingOverride` represents one target override where `Some(Shortcut)`
  remaps the executable shortcut for that resolve and `None` unbinds it for that
  resolve.
- `KeybindingOverrides` is an in-memory, data-only collection with an empty
  default. It must not read from or write to disk, environment variables, host
  state, settings UI, plugin runtime, or Cargo-configured resources.
- A target is "known" if the registered `MenuRegistry` contains that extension
  point and entry id before visibility filtering. Targeting a known but hidden or
  disabled entry must not be reported as unknown; existing visibility and
  enablement behavior still decides whether it appears or executes.
- Overrides apply only to the cloned entries for the current resolve. They must
  not mutate the registry, the default menu definition, or any later normal
  `resolve` result.
- Unbinding removes only that entry's executable `shortcut` from the resolved
  menu entry, accelerator table, `command_for_shortcut`, and
  `enabled_command_for_shortcut` for this resolve. It must not remove the visible
  menu entry, command, label, section/order data, enablement predicate result, or
  display-only `shortcut_hint`.
- Remapping updates the active executable shortcut for that resolved entry and
  for the accelerator/command indexes in that resolve.
- Conflicts introduced by overrides remain non-fatal resolve-time data:
  `ResolveResult::conflicts` reports them deterministically and
  `enabled_command_for_shortcut` suppresses execution for conflicted shortcuts.
- Unknown targets emit deterministic diagnostics and leave the resolved menu tree,
  accelerator count, and conflict set unchanged.
- The shipped default menu is unchanged: default resolve still has exactly 19
  executable accelerators, zero conflicts, and the same entry ids, command ids,
  labels, order, predicates, enablement predicates, shortcut values, and passive
  shortcut hints.
- No host/shell wiring, persistence, settings UI, command-bus behavior, plugin
  runtime behavior, CAD behavior, Cargo/dependency metadata, workflow, schema, or
  automation surface changes.
- The dispatch remains PR-only. Do not take or request a path that fast-forwards
  this production-source change directly to `origin/main`.

## Required Implementation Notes

- Add `crates/editor-ui/src/menus/keybinding.rs` as a pure data module. Use
  standard library collections only; no new dependencies.
- Keep the normal `resolve` implementation as the API existing callers use. It
  may delegate internally to `resolve_with_keybinding_overrides` with
  `KeybindingOverrides::default()` if that keeps equivalence obvious.
- Apply overrides after visibility/label/section/order/enablement resolution has
  produced cloned `ResolvedEntry` values and before the accelerator table,
  shortcut-command index, conflict list, and conflicted-shortcut set are built.
  This preserves registry immutability and lets the returned resolved menu show
  the active remap/unbind state for that resolve.
- Keep `ShortcutConflict` reporting on the existing `ResolveResult::conflicts`
  surface. Add the smallest public resolve diagnostic data needed for unknown
  keybinding targets and tests; keep it under `editor-ui::menus`.
- Preserve `command_for_shortcut` as a binding/display lookup and
  `enabled_command_for_shortcut` as the execution lookup that additionally
  checks enablement and conflict state.

## Behavioral Coverage Required in `menus_ordering.rs`

- Empty overrides preserve default behavior: the default editor menu still
  resolves to 19 executable accelerators and zero conflicts, and normal
  `resolve` equals `resolve_with_keybinding_overrides` with empty overrides for
  the asserted fields.
- Remapping a known target changes that entry's active executable shortcut for
  this resolve only and leaves the registry/default menu unchanged for a later
  normal `resolve`.
- Unbinding a known target removes its executable shortcut from the accelerator
  table and from `command_for_shortcut` / `enabled_command_for_shortcut` for that
  resolve, while keeping the menu entry visible/present when existing
  visibility predicates allow it.
- Remapping a known target onto another live shortcut reports a deterministic
  conflict and suppresses keyboard execution for that conflicted shortcut while
  keeping display/introspection deterministic.
- An unknown target produces a diagnostic without changing the resolved menu
  tree, accelerator count, or conflict set.
