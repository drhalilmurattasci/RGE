### Q3. Current World Handoff API

The handoff API surface that the first consumer can use **without adding a global registry** is the `rge_kernel_ecs::World` value returned by the loader itself, owned as a stack/local in the consumer binary. The relevant kernel-`World` surface already exists.

Type distinction (the Planner notes require it):

It exposes accessors but no constructor that absorbs a pre-built `rge_kernel_ecs::World`; it is therefore not a "receive" surface for a loader-produced kernel world (see below).

Concrete surfaces on `rge_kernel_ecs::World` that produce/consume the loader's output without a global registry (every line citation taken from `kernel/ecs/src/world.rs`):
- `pub fn new() -> Self` at `:75` — the empty-world constructor the loader itself uses internally (`crates/rge-scene-loader/src/lib.rs:85`).
- `pub fn current_tick(&self) -> u64` at `:94`.
- `pub fn last_tick(&self) -> u64` at `:100`.
- `pub fn advance_tick(&mut self)` at `:111-114` — increments `change_tick`, saves prior tick as `last_tick`. This is the tick API the existing golden test already pins against a loader-produced world at `crates/rge-scene-loader/tests/simple_scene.rs:110-127`.
- `pub fn entity_count(&self) -> usize` at `:211`.
- `pub fn entity(&self, entity: EntityId) -> Option<EntityRef<'_>>` at `:221` — used by `crates/rge-scene-loader/tests/simple_scene.rs:62-78` to assert per-entity component presence post-load.

The consumer does not call them directly, but their existence is what lets the loader return a fully-populated `World` value the consumer can simply own.

These public functions on `rge_kernel_ecs::World` form a complete *produce-and-consume* surface for a loader-built world that the runtime can drive without any registry, snapshot system, or scene-graph singleton.

Non-handoff surfaces that the audit must explicitly *rule out* as an integration point (each carries a hard precondition the loader cannot satisfy):

- `EditorShell::with_world_projection_graph(cad_world: KernelWorld, projection: CadProjection, cad_graph: CadGraph) -> Self` at `crates/editor-shell/src/lifecycle/mod.rs:638-655`. **Receives** a `KernelWorld` *by type*, but `:644-654` assert `cad_world` contains *exactly one* `BRepHandle`-carrying entity. The simple-scene loader (Transform/Camera/Light/Visibility only — `crates/rge-scene-loader/src/lib.rs:32-34`) never produces `BRepHandle` entities, so this constructor would panic at handoff. It is not a viable receive API for a loader-produced world.
- `EditorShell::with_world(mut world: World) -> Self` at `crates/editor-shell/src/lifecycle/mod.rs:567-619` takes the **editor-shell wrapper** `World` (`use crate::world::World` at `:208`), *not* `rge_kernel_ecs::World`. The wrapper has no constructor that absorbs an externally built `rge_kernel_ecs::World` (its `kernel_mut()` at `crates/editor-shell/src/world.rs:102-104` only hands out a `&mut` borrow), so this is not a receive API for the loader's output without modifying editor-shell.
- `editor_shell::world::World::kernel()` at `crates/editor-shell/src/world.rs:94-96` / `kernel_mut()` at `:102-104` — these are accessor borrows over a wrapper-owned kernel world, not handoff entry points; they cannot replace the wrapper's kernel field.
- `runtime/runtime-{desktop,mobile,web,headless}/src/main.rs` — all four print-only stubs (each 6 lines, identical shape). They expose no current handoff API; they *are* the surface the first consumer would extend. `runtime-headless` is the smallest such candidate per Q2.

In short: the only handoff path currently justified from current code is "loader returns `rge_kernel_ecs::World`; the consumer owns it directly." No global registry, no snapshot round-trip, no editor-shell extension required for the smallest first consumer.
