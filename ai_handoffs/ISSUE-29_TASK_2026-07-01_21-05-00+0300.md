## Deliverables

- Add `ResolveResult::is_shortcut_bound(&self, shortcut: &Shortcut) -> bool`.
  It is a binding/display pre-check: any visible resolved entry that claims the
  shortcut counts as bound, including disabled and conflicted visible entries.
- Add `ResolveResult::entry_ids_for_shortcut(&self, shortcut: &Shortcut)` as a
  pure read projection over already-resolved data. It must return every
  resolved entry id that claims `shortcut`: the winning id for an unconflicted
  bound shortcut, every `ShortcutConflict::entries` id for a conflicted
  shortcut, and an empty result for an unbound shortcut. The executor may choose
  the exact return type from existing data, but must not add a new
  `ResolveResult` field or expose raw `HashMap` order.
- Add `ResolveResult::shortcuts_for_command(&self, command: &Command) -> impl
  Iterator<Item = &Shortcut>`, returning all effective shortcuts whose winning
  command equals `command` in the same stable order as `bindings()`.
- Add focused `crates/editor-ui/tests/menus_ordering.rs` coverage for bound,
  unbound, disabled-visible, conflicted, and repeated-resolve deterministic
  behavior. Tests must also keep the default menu at 19 executable accelerators
  and zero conflicts.
- Preserve existing `command_for_shortcut`, `shortcut_for_command`,
  `bindings`, `enabled_command_for_shortcut`, `resolve`,
  `resolve_with_keybinding_overrides`, and `apply_keybinding_overrides`
  behavior.
- Perform the required final self-rearm edit in `.ai/dispatch.tasks.md` exactly
  as specified in this packet.

## Acceptance Criteria

- The new helpers are pure read-only accessors over `ResolveResult`; they do
  not mutate the registry, resolved entries, accelerator table, overrides,
  settings, host routing, command routing, or persistence.
- `is_shortcut_bound` returns `true` for visible disabled bindings and visible
  conflicted bindings, and `false` for unbound shortcuts and shortcuts released
  by hidden or predicate-filtered entries.
- `entry_ids_for_shortcut` returns deterministic entry ids for unconflicted and
  conflicted shortcuts, returns an empty result for unbound shortcuts, and does
  not require a new resolver field.
- `shortcuts_for_command` returns every effective shortcut for a command in the
  same stable order as `bindings()` and returns an empty iterator for unbound
  commands.
- Existing resolver behavior remains unchanged: raw override application still
  consumes `keybinding_overrides.iter()` in insertion order, conflict
  collection remains non-fatal resolve-time data, and keyboard execution still
  suppresses disabled or conflicted shortcuts through
  `enabled_command_for_shortcut`.
- Default menu invariants remain unchanged: a full default resolve still has 19
  executable accelerators and zero conflicts.
- No prohibited surface changes; after completion only MAY-edit files and
  current ISSUE-29 handoff/claim artifacts differ.
- `.ai/dispatch.tasks.md` receives exactly one new live marker with reason text
  exactly `select the next bounded FEATURE surface for task 182; post-feature
  audit cadence remains retired.`, plus a concise `Recommendation for human
  approval` block, and no task 182 is appended.
