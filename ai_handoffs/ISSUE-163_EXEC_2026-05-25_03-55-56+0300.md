## 5-Question Typed ComponentValue Bridge Preflight Answer Block

### Q1. Bridge owner plus dependency direction

Answer: No existing crate currently owns the full typed `rge_data::Scene` to
`rge_kernel_ecs::World` bridge. Current dependency evidence rules out several
candidates and constrains direction, but it does not name an existing owner.

Current-code anchors confirmed by inspection:

- `rge-data` is documented as a stateless serde plus migration layer
  (`crates/rge-data/src/lib.rs:13`) and exports the schema envelope types only
  (`crates/rge-data/src/lib.rs:62-68`). Its scene module says component payloads
  are reflection-neutral and that downstream callers later walk
  `ComponentValue::data` through `kernel/types::serde_bridge::from_ron` keyed
  by `ComponentValue::type_id` (`crates/rge-data/src/scene.rs:14-22`,
  `crates/rge-data/src/scene.rs:89-95`).
- `rge-data`'s dependency floor is `rge-kernel-asset`, `serde`, `ron`, `ulid`,
  `thiserror`; the only ECS edge is a test-only `rge-kernel-ecs` dev-dependency
  (`crates/rge-data/Cargo.toml:12-20`). It does not depend on the component
  crates or on `kernel/types`.
- Component crates each depend only on `serde` plus their local needs
  (`crates/components-spatial/Cargo.toml:12-16`,
  `crates/components-render/Cargo.toml:12-17`,
  `crates/components-visibility/Cargo.toml:12-16`). No component crate depends
  on `rge-kernel-ecs` or on `rge-data` today.
- Forbidden-dep rule 6 forbids renderer crates `rge-gfx`, `rge-gfx-ir`, and
  `rge-brep-render` from depending on game-domain crates
  (`tools/architecture-lints/src/forbidden_dep.rs:38-39`,
  `tools/architecture-lints/src/forbidden_dep.rs:173-184`). `rge-components-*`
  is classified game-domain via the `GAME_DOMAIN_PREFIXES` list
  (`tools/architecture-lints/src/forbidden_dep.rs:45-52`,
  `tools/architecture-lints/src/forbidden_dep.rs:446`).

Plausible candidates and dependency implications:

- `crates/rge-data`: would have to add `rge-data -> rge-components-spatial`,
  `rge-data -> rge-components-render`, `rge-data -> rge-components-visibility`,
  `rge-data -> rge-kernel-ecs`, and almost certainly `rge-data -> rge-kernel-types`.
  This contradicts the documented stateless-schema posture
  (`crates/rge-data/src/lib.rs:13`) and the module's own comment that downstream
  callers do the typed bridging (`crates/rge-data/src/scene.rs:14-22`). Not a
  valid owner without a substrate decision.
- `kernel/types` or `kernel/ecs`: cannot own the bridge because Tier-1 kernel
  crates cannot depend on Tier-2 crates under forbidden-dep rule 1
  (`tools/architecture-lints/src/forbidden_dep.rs:6-7`,
  `tools/architecture-lints/src/forbidden_dep.rs:119-128`). The bridge needs
  `rge-data` plus the component crates, all Tier 2.
- A new `kernel/scene-bridge` cavity: same Tier-1 problem as other kernel
  crates if it depends on `rge-data` or component crates. That would be a new
  substrate-level decision, not current code.
- A new Tier-2 `rge-scene-loader` (or equivalent) crate: could depend on
  `rge-data`, `rge-kernel-ecs`, `rge-kernel-types`, and the component crates
  without tripping rule 1 or rule 6, and would preserve `rge-data`'s schema-only
  posture. This is the most plausible direction, but no such crate exists today.
- `runtime/runtime-headless` and `runtime/runtime-desktop`: currently carry an
  empty `[dependencies]` section
  (`runtime/runtime-headless/Cargo.toml:16-17`,
  `runtime/runtime-desktop/Cargo.toml:16-17`). They could depend on the needed
  crates without tripping rule 6, but ownership here would duplicate bridge
  logic per runtime binary unless a reusable owner is introduced.
- `crates/editor-shell`: wraps a `rge_kernel_ecs::World`
  (`crates/editor-shell/src/world.rs:53-63`) and exposes `kernel_mut()` for
  typed insertion (`crates/editor-shell/src/world.rs:98-104`). It does not
  currently depend on `rge-data` or on any `rge-components-*` crate. That makes
  it editor-only — usable as a typed sink for editor-driven loads, but not as a
  general runtime/cook bridge owner.
- `editor/rge-editor`: editor crates are not on the rule-6 renderer list, but
  editor ownership again limits the bridge to editor-time usage and bypasses
  cook / dedicated-server callers (`runtime/runtime-headless`).
- The component crates (`rge-components-spatial`, `rge-components-render`,
  `rge-components-visibility`): each crate could own impls for its own types
  under Rust's orphan rule, but no single component crate can own the whole
  scene bridge spanning every component family without importing `rge-data` and
  the other component crates — adding cross-component-crate edges that do not
  exist today.
- Renderer crates `rge-gfx`, `rge-gfx-ir`, `rge-brep-render`: forbidden by
  rule 6 (`tools/architecture-lints/src/forbidden_dep.rs:38-39`,
  `tools/architecture-lints/src/forbidden_dep.rs:173-184`).
- `crates/script-host`: explicitly hard-codes a `Counter` ECS bridge and
  defers the generic component bridge to "Phase 4-Foundation"
  (`crates/script-host/src/ecs_bridge.rs:1-8`,
  `crates/script-host/src/ecs_bridge.rs:33-44`). Not a scene loader owner.
- A test-local helper: the current `rge-data` load+tick test ships an
  identity-only bridge that spawns ECS entities and deliberately ignores
  component payloads
  (`crates/rge-data/tests/golden_simple_scene_load_tick.rs:1-17`,
  `crates/rge-data/tests/golden_simple_scene_load_tick.rs:50-57`). It is a test
  fixture, not a production architecture surface.

Conclusion: existing dependency rules justify several "must not live here"
answers (kernel crates, renderer crates, script-host, plain `rge-data`) and
strongly suggest that any production bridge must sit above the schema, kernel,
and component crates as a new Tier-2 loader or as an explicitly assigned
runtime/editor owner. They do not justify a concrete existing owner.

### Q2. type_id to component-type mapping mechanism

Answer: No current production mechanism maps `ComponentValue.type_id` strings
to concrete Rust component types.

`ComponentValue` stores `type_id: String` and `data: String`
(`crates/rge-data/src/scene.rs:89-95`). `kernel/types::TypeId` is a 128-bit
hash generated from a fully qualified type name via `TypeId::of_name`
(`kernel/types/src/type_id.rs:48-73`). `Reflect` exposes `TYPE_ID`,
`TYPE_NAME`, `FQ_TYPE_NAME`, `SCHEMA_VERSION`, `FIELDS`, and `KIND` as
associated consts plus `get_field_dyn` / `set_field_dyn`
(`kernel/types/src/reflect.rs:60-108`). The `serde_bridge` helpers are
generic functions, not keyed dispatch: `to_ron<T: Reflect + Serialize>` and
`from_ron<T: Reflect + DeserializeOwned>` both require the concrete `T` at
each call site (`kernel/types/src/serde_bridge.rs:117-135`).

The reflect crate explicitly forbids a global runtime registry: "No global
registry at runtime. Each reflected type exposes its `FieldDescriptor` slice
as a `&'static` constant. A future `inventory!` crate is explicitly out of
scope" (`kernel/types/src/lib.rs:38-46`). The derive macro mirrors that —
it emits one `impl Reflect` and no registry submission
(`crates/macros-reflect/src/codegen.rs:1-6`). It composes
`FQ_TYPE_NAME` from `module_path!()` plus the source name
(`crates/macros-reflect/src/codegen.rs:96-98`,
`crates/macros-reflect/src/codegen.rs:112`) but emits `TYPE_ID` as
`TypeId::from_bytes([0u8; 16])` (`crates/macros-reflect/src/codegen.rs:108-110`).
It also rejects enums and non-named structs today
(`crates/macros-reflect/src/derive.rs:32-50`).

The four simple-scene component types do not derive `Reflect` (confirmed by a
repo-wide `derive(... Reflect ...)` grep over all three component crates,
which returned no matches). Their current derive lists are:

- `Transform`: `Debug, Clone, Copy, PartialEq, Serialize, Deserialize`
  (`crates/components-spatial/src/transform.rs:22-23`).
- `Camera`: `Debug, Clone, Copy, PartialEq, Serialize, Deserialize`
  (`crates/components-render/src/camera.rs:43-44`); nested `Projection` enum
  has the same derive surface
  (`crates/components-render/src/camera.rs:8-9`).
- `Light`: `Debug, Clone, Copy, PartialEq, Serialize, Deserialize`
  (`crates/components-render/src/light.rs:56-57`); nested `LightKind` enum has
  the same derive surface (`crates/components-render/src/light.rs:18-19`).
- `Visibility`: `Debug, Default, Clone, Copy, PartialEq, Eq, Hash, Serialize,
  Deserialize` (`crates/components-visibility/src/visibility.rs:19-20`).

The broader public component-crate surface that a bridge would have to cover
also lacks `Reflect`. Spatial exports `ChildOf`, `Entity`, `GlobalTransform`,
`Parent`, `Transform` (`crates/components-spatial/src/lib.rs:32-36`). Render
exports `AssetId`, `NULL_ASSET_ID`, `BRepHandle`, `CadNodeId`, `Camera`,
`Projection`, `Light`, `LightKind`, `Lod`, `LodLevel`, `MaterialHandle`,
`MeshHandle`, `ReflectionProbe`, `SkinnedMesh`
(`crates/components-render/src/lib.rs:31-40`). Visibility exports `Disabled`,
`Hidden`, `Highlight`, `Visibility`
(`crates/components-visibility/src/lib.rs:25-28`). Inspected public structs
and enums in these crates derive serde-oriented traits but never `Reflect`,
`Component`, or `SnapshotComponent`
(`crates/components-spatial/src/child_of.rs:14-15`,
`crates/components-spatial/src/entity.rs:14-16`,
`crates/components-spatial/src/global_transform.rs:17-18`,
`crates/components-spatial/src/parent.rs:17-19`,
`crates/components-render/src/brep_handle.rs:15-16`).

The `rge-data` schema test pins canonical strings for the four typed
components (`crates/rge-data/tests/golden_simple_scene_schema.rs:24-27`):

- `rge::components::Transform`
- `rge::components::Camera`
- `rge::components::Light`
- `rge::components::Visibility`

These are fixture-shape assertions, not a production registry or dispatch
table; the TASK packet explicitly forbids treating them as one. The TASK
packet also forbids treating `serde_bridge::from_ron<T>` as a keyed dispatch
mechanism, and the inspected signature confirms it is a single-type generic
helper (`kernel/types/src/serde_bridge.rs:132-135`).

Therefore current code provides:

- The canonical type-id string convention (`rge::components::*`), only in a
  schema fixture test.
- A reflection contract whose `FQ_TYPE_NAME` could match those strings IF the
  component types adopted `#[derive(Reflect)]` AND lived under matching module
  paths — but none of the four types derive `Reflect`, and the macro emits
  `TYPE_ID` as zero bytes today.
- A generic `from_ron<T>` that needs the concrete `T` at the call site.

No current code maps an arbitrary `ComponentValue.type_id` string to a Rust
type and dispatches `from_ron` for that type.

### Q3. ECS insertion target

Answer: The current ECS target for entity identity is clear; typed component
attachment for the four scene component types is not usable as-is.

Entity identity:

- `rge_data::EntityId` is a transparent ULID wrapper
  (`crates/rge-data/src/entity_ref.rs:35-44`).
- `rge_kernel_ecs::EntityId` also wraps a ULID and exposes
  `EntityId::from_ulid` (`kernel/ecs/src/entity.rs:21-46`).
- `World::spawn_with_id(id: EntityId)` preserves the ID supplied by the
  caller (`kernel/ecs/src/world.rs:127-153`).
- The current load+tick test bridges `rge_data::EntityId` to
  `rge_kernel_ecs::EntityId::from_ulid(entity.id.0)` and calls
  `world.spawn_with_id(id)` (`crates/rge-data/tests/golden_simple_scene_load_tick.rs:50-57`).

Typed component insertion surface on `World`:

- `World::insert<C: Component>(entity, component)`
  (`kernel/ecs/src/world.rs:243-252`).
- `World::spawn_with<C: Component>(component) -> EntityId`
  (`kernel/ecs/src/world.rs:177-183`).
- `EntityMut::insert<C: Component>(component)`
  (`kernel/ecs/src/entity.rs:163-166`).
- Free-function aliases `insert<C: Component>` and `insert_component<C: Component>`
  (`kernel/ecs/src/lib.rs:54-83`).

Erased insertion path:

- `World::insert_erased(entity, value: Box<dyn Any + Send + Sync>, type_id: TypeId)`
  is `pub(crate)` — visible only inside `rge-kernel-ecs`
  (`kernel/ecs/src/world.rs:155-175`). The snapshot restore path is its only
  caller (`kernel/ecs/src/snapshot.rs:347-351`).

`Component` trait shape:

- `pub trait Component: Any + Send + Sync + 'static {}` with no derive macro;
  the doc explicitly says "Implement manually"
  (`kernel/ecs/src/component.rs:9-20`).
- `SnapshotComponent: Component + Serialize + DeserializeOwned` and requires
  opt-in registration via `World::register_snapshot_component<C>`
  (`kernel/ecs/src/snapshot.rs:49-76`,
  `kernel/ecs/src/snapshot.rs:159-173`).

The four scene component types do not implement `rge_kernel_ecs::Component` or
`SnapshotComponent` (confirmed by repo-wide grep for
`impl ... Component for (Transform|Camera|Light|Visibility)` — the only hit is
`kernel/ecs/tests/smoke_100k.rs:13`, which impls `Component` for a test-local
3-field struct named `Transform`, not `rge_components_spatial::Transform`).

A third bridge crate cannot add `impl Component` for these external types
because of Rust's orphan rule; those impls must live in the owning component
crates or use new wrapper types. The public `insert<C: Component>` family
therefore cannot accept the four component types today, and the erased
insertion path is private (`pub(crate)`) and gated by `SnapshotComponent`
registration. Both routes are new source design outside this read-only audit.

### Q4. Justified-from-current-code check

Answer: NO.

The bridge owner plus dependency direction is only partially constrained by
current code: `rge-data`, kernel crates, renderer crates, and `script-host` are
not viable owners under forbidden-dep rule 1, forbidden-dep rule 6, `rge-data`'s
documented schema-only posture, and `script-host`'s explicit
generic-bridge-deferred prototype scope. A higher Tier-2 loader, runtime owner,
or editor owner is plausible, but no existing crate currently owns the full
bridge as-is.

The type-id mapping mechanism is not justified by current code. The repo has
canonical schema-string constants pinned only in a `rge-data` fixture test,
`TypeId`, `Reflect`, generic `serde_bridge` helpers, and macro-emitted
`FQ_TYPE_NAME`, but no registry, match table, inventory submission, or existing
`#[derive(Reflect)]` on the four component types that would let a caller turn
`ComponentValue.type_id` into a concrete Rust type. The ECS insertion step also
requires the target types to implement `rge_kernel_ecs::Component`, which none
of the four scene component types do today, and the orphan rule prevents a
third crate from adding those impls.

Because both the bridge owner decision and the type-id mapping mechanism are
not current-code surfaces, Q4 cannot be `YES`.

### Q5. Smallest implementation dispatch or NEEDS_HUMAN

Answer: NEEDS_HUMAN.

Specific arbiter decisions required before any implementation dispatch is
safe:

1. Bridge owner and dependency direction. Choose between a new Tier-2
   scene-loader crate (e.g. `rge-scene-loader`), a runtime-owned bridge inside
   `runtime/runtime-headless` and `runtime/runtime-desktop`, an editor-owned
   bridge wired through `crates/editor-shell`, or another explicit owner.
   Forbidden-dep rule 1 and rule 6 plus `rge-data`'s schema-only posture rule
   out kernel ownership, renderer ownership, and ownership inside `rge-data`
   itself.
2. Type-id mapping mechanism. Choose between an explicit registry / match
   table in the bridge owner, a `Reflect`-driven mapping (which requires adding
   `#[derive(Reflect)]` to the four component types AND deciding whether
   `FQ_TYPE_NAME` may diverge from the existing `rge::components::*` schema
   strings or whether the macro/schema must be aligned), a hand-rolled
   `type_id` → typed-`from_ron::<C>` dispatch, or another approved design. The
   `kernel/types/src/lib.rs:38-46` "no global registry" rule must be
   respected.
3. How scene component types become ECS components. Choose between adding
   `impl rge_kernel_ecs::Component` (and possibly `SnapshotComponent`) in the
   owning component crates `rge-components-spatial`, `rge-components-render`,
   `rge-components-visibility` (the only crates that can satisfy the orphan
   rule for these types), introducing wrapper types in the bridge owner, or a
   different attachment strategy entirely.

No implementation dispatch is safe before those decisions, because current code
does not provide the bridge owner or the mapping mechanism as-is, and the four
typed component types would each need either a new trait impl in their owning
crate or a wrapper-type surface to be attached to `rge_kernel_ecs::World`.

Recommendation: NEEDS_HUMAN.
