## What I Changed

### Source

- `Invoke-AiDispatchQueue.ps1`:
  - Added a new helper `Get-FailureTaxonomyLabels` immediately before the
    `# --- Environment ---` section. It is a pure text classifier over the
    queue-local outcomes `$LoopText`, `$ExecStatus`, and `$PublishHardFailed`;
    it does not read or write files, call git/gh, or alter loop output. It
    always returns at least one taxonomy label (defaulting to
    `ai-dispatch-failure-unknown`).
  - Extended the existing idempotent `$labelSpec` list inside the bookkeeping
    label setup block to also create the seven new failure-class labels
    (`stall`, `timeout`, `blocked`, `verification`, `control`, `publish`,
    `unknown`) through the same `gh label create ... --force` path that
    creates the existing queue labels.
  - After `$willRetry` is computed in the post-publish finalization block,
    added a `$taxonomyLabels` computation guarded by
    `$runFailed -and -not $willRetry`, so taxonomy is only computed for
    terminal failures.
  - Extended the non-retry relabel command builder so it appends one
    `--add-label <taxonomy>` argument per selected taxonomy label. The retry
    relabel branch and the success path are untouched.
  - Extended the post-relabel label finalization verification: only when
    `$runFailed` is true, every selected taxonomy label must also be present
    on the issue. A missing taxonomy label flips `$labelOk` to `$false`,
    which causes the existing `Fail` path to exit non-zero — matching the
    existing `ai-dispatch-failed` finalization behavior.

## Open Questions for Reviewer

- Verification timeouts: the classification order in the task (publish ->
  blocked -> stall -> timeout -> verification -> control -> unknown) means
  a `Verification timed out` loop-text line is classified as
  `ai-dispatch-failure-timeout` rather than `ai-dispatch-failure-verification`,
  because the timeout rule fires before the verification rule. This matches
  the literal ordering in the TASK packet and the acceptance criterion
  "Generic timeouts that are not Codex watchdog stalls are classified as
  `ai-dispatch-failure-timeout`." Flagging it explicitly so the reviewer
  can confirm the intent.
- The `(?i)maxcorrectionrounds=\d+ is exhausted` clause in the control rule
  is intentionally only reached after the verification rule, so the
  Verification gate's `"Verification gate failed (exit X) and
  MaxCorrectionRounds=N is exhausted."` line classifies as
  `verification`, while the control gate's `"Codex requested changes, but
  MaxCorrectionRounds=N is exhausted."` line classifies as `control`. Both
  fall through to the more specific anchor (`verification gate failed` and
  `codex requested changes`) first; the `maxcorrectionrounds=...exhausted`
  clause is a defense-in-depth catch for log truncation. Confirming the
  reviewer is happy with this layering.
