## Goal

Make `Invoke-AiDispatchQueue.ps1` run each selected issue in an isolated git
worktree by default, so the primary checkout remains on clean `main` while the
Codex/Claude dispatch loop edits, verifies, and commits on
`ai-dispatch/ISSUE-$IssueNumber`. This is the single-dispatch queue path only; preserve
the existing queue lock, publish modes, retry/label taxonomy, progress/result
comments, and issue-close behavior. This packet also incorporates the prior
Codex control feedback for ISSUE-231: durable failure, interruption, and orphan
recovery reporting must include the preserved or archived worktree path and
inspection/removal commands in stdout, dispatch audit logs, and GitHub result or
recovery comments.

## Requirements

1. Keep the primary checkout on `main` during inner-loop execution. The queue
   may sync and validate the primary checkout before creating the worktree, but
   it must not check out `ai-dispatch/ISSUE-$IssueNumber` in the primary checkout for the
   dispatch run.
2. For each selected issue, create an isolated worktree for
   `ai-dispatch/ISSUE-$IssueNumber` from the synced `main` or `origin/main` base. Use a
   deterministic sibling path outside the repo root, following the documented
   `dispatch-worktrees` convention: a directory named `dispatch-worktrees` next
   to the repo, with a child directory named for the issue such as `ISSUE-231`.
3. Run `Invoke-AiDispatchLoop.ps1`, scope guard checks, `git add`, `git commit`,
   dispatch-log generation, publish-mode preparation, and any branch-status
   reads from the isolated worktree. Use explicit paths, `git -C`, or tightly
   scoped `Set-Location` in `try`/`finally` so no command accidentally operates
   on the primary checkout.
4. Preserve publish semantics from ISSUE-230:
   - `main` mode fast-forwards and pushes `origin/main` only after loop exit 0
     and Codex control verdict `pass`.
   - `branch` mode and `-NoPublish` leave the dispatch branch for human review,
     without pushing.
   - `pr` mode pushes the dispatch branch and opens a PR, without merging,
     without pushing `origin/main`, and without closing the source issue.
5. Preserve terminal label reconciliation, retry eligibility, failure taxonomy,
   progress comments, result comments, and final issue-close behavior except
   where worktree isolation requires adding path/status details.
6. On successful queue completion, safely remove the isolated worktree only
   after the dispatch commit is preserved on its branch and the selected publish
   action has completed. Do not remove a worktree with uncommitted changes or
   an unpreserved branch commit. The primary checkout should end clean, on
   `main`, and in sync with `origin/main` when a remote exists.
7. On failed, blocked, interrupted, or orphan-recovery paths, preserve useful
   worktree state for human inspection. If an existing deterministic worktree
   path must be archived to a `.interruptN` path, report the archived path. If
   the worktree remains in place, report the preserved path.
8. Durable worktree-status reporting must be consistent. The local stdout
   status text, dispatch audit log body, final GitHub result comment, and
   orphan-recovery comments must include the relevant deterministic, preserved,
   or archived worktree path plus practical commands such as:
   - `git -C $WorktreePath status --short --branch`
   - `git -C $WorktreePath log --oneline -5`
   - `git worktree remove $WorktreePath` only when it is safe and appropriate
     to tell a human how to remove that worktree
9. `Write-DispatchLog` must write the worktree path/status into the audit log
   content when a worktree is preserved, archived, or removed; using the
   worktree root only for log location or git scope is not sufficient.
10. Orphan-recovery comments must include the path returned or selected by the
    recovery/archive operation, including `.interruptN` archive paths produced
    by `Save-OrphanDispatchWorktree`.
11. Add focused Pester coverage for any new pure helpers used to compute
    worktree paths, decide cleanup/preserve behavior, or format worktree status
    text. Coverage must explicitly pin interrupted/orphan recovery reporting so
    a `.interruptN` archive path or preserved worktree path appears in the
    durable comment/status text.
12. Do not implement multi-dispatch fan-out, parallel scheduling, queue lock
    changes, autonomous task-selection changes, or inner-loop changes.

## Deliverables

- `Invoke-AiDispatchQueue.ps1` creates and uses an isolated worktree for the
  selected issue branch instead of switching the primary checkout to the
  dispatch branch.
- Queue commit, dispatch-log, scope-guard, publish, result-comment, retry, and
  cleanup paths operate against the isolated worktree or an explicit worktree
  root.
- Failed, blocked, interrupted, and orphan-recovery paths preserve useful
  worktree state and report the preserved or archived path in durable artifacts.
- Focused Pester tests cover worktree path/status/cleanup helper behavior and
  interrupted/orphan recovery reporting.
- `AI_DISPATCH_AUTOMATION.md` documents the new default queue worktree
  isolation, success cleanup behavior, and failure/recovery inspection path.
- `AI_DISPATCH_PARALLEL.md` is updated only if a short cross-reference is
  needed to keep the serial queue and parallel docs consistent.
- The ISSUE-231 EXEC packet and any ISSUE-231 queue audit log are generated in
  the normal dispatch flow.

## Acceptance Criteria

- A queue dispatch no longer switches the primary checkout to
  `ai-dispatch/ISSUE-$IssueNumber` during the inner loop.
- Dispatch artifacts and code changes are committed from the isolated worktree
  onto `ai-dispatch/ISSUE-$IssueNumber`.
- After a successful queue run, the primary checkout is clean `main` and synced
  with `origin/main` when that remote exists.
- `branch`, `main`, and `pr` publish modes keep their ISSUE-230 behavior.
- Failure, blocked, interrupted, and orphan-recovery paths preserve enough
  worktree state for human recovery and report the preserved or archived
  worktree path in stdout, dispatch logs, and GitHub comments.
- Orphan-recovery comments include the `.interruptN` archive path or preserved
  worktree path after `Save-OrphanDispatchWorktree` runs.
- `Write-DispatchLog` includes deterministic worktree-status text in the audit
  log body for preserved, archived, or removed worktrees.
- Focused Pester tests pin worktree path computation, cleanup/preserve
  decisions, status formatting, and interrupted/orphan recovery reporting.
- Documentation accurately describes default worktree isolation, success
  cleanup, and failure/recovery inspection and removal behavior.
- No forbidden files are modified or added.
- The canonical verification gate passes.

## Halt Conditions

The Executor MUST halt without committing and request fresh guidance if any of
the following occur:

- Implementing isolation requires editing a file outside the MAY-edit set.
- The queue would need to modify `Invoke-AiDispatchLoop.ps1`,
  `Invoke-AiDispatchAuto.ps1`, scheduler scripts, workflows, schemas, packet
  templates, `.ai/dispatch.verify.ps1`, `.ai/dispatch.tasks.md`, or production
  Rust/source files.
- A safe deterministic worktree path outside the repo root cannot be computed.
- A pre-existing deterministic worktree path contains useful uncommitted or
  unpreserved work and cannot be safely archived or reported without data loss.
- A failure, interruption, or orphan-recovery path cannot report the preserved
  or archived worktree path in durable output.
- Preserving ISSUE-230 `main`, `branch`, `pr`, or `-NoPublish` behavior would
  require a semantic change rather than a worktree-root plumbing change.
- Focused Pester coverage would require live `gh`, `codex`, `claude`, network
  access, a nested queue dispatch, or destructive operations outside a temp
  fixture.
- The canonical verification gate fails for reasons that require edits outside
  this packet's scope to fix.
