## Goal

Change the dispatch tooling's default publish posture to human-reviewed PR publication. A queue run without a publish flag must no longer default to fast-forwarding and pushing `origin/main`; the mechanical default across the queue, autonomous driver, and autonomous scheduler should be `pr`, while explicit `main`, explicit `branch`, and legacy `-NoPublish` behavior remain available and tested.

## Deliverables

- `Invoke-AiDispatchQueue.ps1` resolves no-flag queue publish mode to `pr`.
- `Invoke-AiDispatchQueue.ps1 -PublishMode main` still selects main auto-publish mode.
- `Invoke-AiDispatchQueue.ps1 -PublishMode branch` and legacy `-NoPublish` still select branch-only mode.
- Conflicting queue arguments remain fail-fast, specifically `-NoPublish -PublishMode main` and `-NoPublish -PublishMode pr`.
- `Invoke-AiDispatchAuto.ps1` defaults `-PublishMode` to `pr` while still accepting explicit `branch`, `main`, and `pr`.
- `Register-AiDispatchSchedule.ps1 -Autonomous` defaults to scheduling autonomous dispatch in PR mode unless the caller explicitly passes another publish mode.
- Existing PR publish behavior remains intact: push the dispatch branch, open a PR targeting `main`, use `Refs #<n>`, do not push `origin/main`, and do not close the source issue automatically.
- Focused Pester coverage pins the queue publish-mode default and the auto/scheduler default contract without invoking live GitHub operations or registering a real scheduled task.
- `AI_DISPATCH_AUTOMATION.md` concisely documents the new defaults and no longer describes no-flag queue runs as main auto-publish.
- The EXEC packet reports the exact files changed and the required gate results.

## Acceptance Criteria

1. A no-flag `Invoke-AiDispatchQueue.ps1` run resolves to `pr` publish mode, not `main`.
2. `Invoke-AiDispatchQueue.ps1 -PublishMode main` still explicitly selects main auto-publish mode.
3. `Invoke-AiDispatchQueue.ps1 -PublishMode branch` and legacy `-NoPublish` still select branch-only mode.
4. Conflicting arguments remain fail-fast: `-NoPublish -PublishMode main` and `-NoPublish -PublishMode pr` are invalid.
5. `Invoke-AiDispatchAuto.ps1` defaults `-PublishMode` to `pr`, while still accepting explicit `branch`, `main`, and `pr`.
6. `Register-AiDispatchSchedule.ps1 -Autonomous` defaults to scheduling PR-mode autonomous dispatch unless the caller explicitly passes another publish mode.
7. PR mode behavior itself is not redesigned: it still pushes the dispatch branch, opens a PR, uses `Refs #<n>`, does not push `origin/main`, and does not close the source issue automatically.
8. Focused Pester coverage pins the queue publish-mode default and the auto/scheduler default contract.
9. `AI_DISPATCH_AUTOMATION.md` matches the implementation defaults and no longer describes no-flag queue as main auto-publish.
10. No files outside the MAY edit / MAY add lists are changed or created.

## Planner Notes

This is a dispatch-tooling safety task. The current operator posture is that successful AI dispatch work should go through a PR by default for human review; forgetting a flag must not silently fast-forward and push `origin/main`.

The likely narrow implementation path is to update the queue publish-mode normalization default from `main` to `pr`, update `Invoke-AiDispatchAuto.ps1` and `Register-AiDispatchSchedule.ps1` parameter defaults from `branch` to `pr`, revise help text/comments/docs that describe the defaults, and extend the existing Pester tests under `tools/dispatch-tests/` to assert those contracts. Keep tests side-effect-free by using the existing queue dot-source seam and PowerShell AST inspection for outer scripts where appropriate.

Run this issue through `Invoke-AiDispatchQueue.ps1 -PublishMode pr` explicitly. This dispatch changes the default, but this dispatch itself must not rely on no-flag queue behavior.
