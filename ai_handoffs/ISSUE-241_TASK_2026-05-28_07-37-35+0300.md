## Goal

Harden `Wait-GitHubActions.ps1` so it can, when explicitly requested, accept PR-side external check evidence such as org/default CodeQL from the pull request check rollup for the exact commit being waited on. The default waiter behavior must remain strict for tracked repository workflows discovered through `gh run list`, and the fix must not introduce a recent-run or unrelated-branch fallback.

## Deliverables

- An explicit opt-in path in `Wait-GitHubActions.ps1` for accepting named PR-side external checks for a specified pull request.
- Strict PR-head-SHA validation: PR-side evidence is usable only when the PR check rollup is for the same head commit SHA as the script's `-Commit` argument.
- Exit-code behavior preserved: `0` only when all expected checks pass, `1` for reached failing conclusions, and `2` for timeout or unresolved missing checks.
- Focused side-effect-free Pester coverage under `tools/dispatch-tests/**`.
- Concise documentation in `AI_DISPATCH_AUTOMATION.md` only if the new option needs operator-facing explanation.
- Generated `ISSUE-241` execution packet and queue audit log only as produced by the dispatch tooling.

## Acceptance Criteria

- Default `Wait-GitHubActions.ps1` behavior remains strict and unchanged for tracked workflows: expected workflow names are resolved from `gh run list` for the requested commit and branch, and a missing expected workflow does not silently pass.
- The script exposes an explicit call-site opt-in for PR-side checks, for example `-PullRequest 240` plus `-AcceptPrSideCheck CodeQL`, or an equivalent clear parameter pair. This relaxation must not be default behavior.
- With the opt-in supplied, a missing `gh run list` workflow may be treated as passed only if PR-side check data for the specified PR shows the explicitly named check completed successfully for the exact same SHA supplied to `-Commit`.
- The implementation must not accept a green check because it passed recently, passed on another commit, or appears on an unrelated branch or workflow run.
- Failing, cancelled, skipped, timed-out, missing, or otherwise non-success PR-side checks must not be accepted as pass.
- A PR-side check rollup whose PR head SHA differs from `-Commit` must not be accepted as pass.
- Exit code `0` is returned only when every expected tracked workflow passed and every explicitly accepted PR-side external check passed under the strict PR-head-SHA semantic.
- Exit code `1` is returned when an expected tracked workflow or accepted PR-side external check reaches a failing conclusion.
- Exit code `2` is returned on timeout or unresolved missing checks, including missing external PR-side checks.
- Tests cover tracked workflow success, default strict failure or timeout for a missing tracked workflow, accepted PR-side success with a matching PR head SHA, rejection with a mismatched PR head SHA, and rejection of failing or missing PR-side checks.
- Tests are side-effect-free: no live GitHub calls, branch pushes, issue or PR mutations, or dependence on repository state outside fixtures and mocks.
- If `AI_DISPATCH_AUTOMATION.md` is touched, the edit is concise and limited to the new waiter option or the PR-side CodeQL visibility behavior.
