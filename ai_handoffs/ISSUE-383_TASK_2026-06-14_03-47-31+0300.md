## Deliverables

- `ProjectedCommandPaletteEntry` or an equivalent host-local palette row model exposes deterministic conflict detail for enabled command-palette rows whose displayed shortcut exactly matches one `ProjectedMainMenu.conflicts` shortcut.
- Conflict detail includes the ordered peer entry ids from the matching `ProjectedShortcutConflict.entries` vector and does not synthesize or re-resolve peer ids from registries.
- Unconflicted enabled rows expose no conflict detail.
- Disabled rows remain visible but non-selectable/non-activatable, and disabled rows do not gain conflict detail from this annotation.
- The command-palette UI renders the conflict detail as informational row text or an equally stable row state while preserving row activation behavior.
- Focused unit tests cover conflicted, unconflicted, and disabled palette rows, plus unchanged ordering and activation behavior.
- Status/task docs mark task 129 complete and do not append task 130.

## Acceptance Criteria

- `command_palette_entries(&ProjectedMainMenu)` uses only the already-projected `main_menu.conflicts` data to derive palette conflict detail.
- Conflict matching is by displayed shortcut string and only for `entry.enabled == true`.
- Peer ids are copied from `ProjectedShortcutConflict.entries` preserving their existing order.
- Filtering, fuzzy scoring, blank-filter pinned/recent promotion, non-blank score ordering, keyboard selection, Enter activation, mouse activation, Pin/Unpin, search focus, Shortcut Help, and Shortcut Conflicts behavior remain unchanged.
- Existing menu command dispatch continues to return/enqueue the same `Command` values for palette activation; conflict annotation is not a block, reroute, or remap.
- `crates/editor-egui-host/src/lib.rs` is edited only if the existing module/import visibility requires it.
- `.ai/dispatch.tasks.md` has no new `130.` heading after the implementation.
