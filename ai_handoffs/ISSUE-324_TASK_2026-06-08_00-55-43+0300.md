## Planner Notes

The current `restore_from_snapshot` body already has the relevant branch: `name_to_fns.get(name) == None` emits `tracing::warn!` and continues. A narrow implementation is to factor restore through one helper that accepts an optional diagnostics sink, or to make the new diagnostics-aware method contain the shared implementation while the old method delegates with a no-op sink. Preserve the tracing warning if that is part of the existing compatibility behavior.

The root workspace already declares `rge-kernel-diagnostics`; if ECS needs direct access to `DiagnosticSink`, add `rge-kernel-diagnostics = { workspace = true }` to `kernel/ecs/Cargo.toml` only. The diagnostics crate exposes `Diagnostic`, `DiagnosticSink`, `Severity`, and `FailureClass::SnapshotRecoverable`; no diagnostics-crate changes should be necessary.

The most important test is an otherwise valid snapshot containing two registered component types restored into a target world that registers only one of them. That test should prove the registered component restores, the unregistered component is absent, and the skipped component is represented by a structured warning diagnostic.
