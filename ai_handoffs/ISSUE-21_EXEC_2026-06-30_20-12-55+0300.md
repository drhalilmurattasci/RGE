## Audit Findings

### Accessor implementation

- `ResolveResult::shortcut_for_command` and `ResolveResult::bindings` are
  present in `crates/editor-ui/src/menus/registry.rs` as methods on the existing
  `ResolveResult`. No `mod.rs` export or new public type was needed.
- `ResolveResult` still has a single private `shortcut_commands:
  HashMap<Shortcut, (Command, bool)>` field. I found no new field, secondary
  index, cached vector, global state, or resolve-time side table.
- `shortcut_for_command` calls `self.bindings()` and returns the first matching
  command in the same stable order as `bindings()`.
- `bindings()` reads `self.shortcut_commands.iter()`, maps it to a local
  `Vec<(&Shortcut, &Command)>`, sorts that local vector, and returns
  `bindings.into_iter()`. It does not expose raw `HashMap` iteration order.
- The ordering rule is documented: shortcut display string first, then
  deterministic structural tie-breakers. The multi-shortcut-to-one-command rule
  is documented: `shortcut_for_command` returns the first item in `bindings()`
  stable order.
- The helper `stable_shortcut_binding_cmp` sorts by `Shortcut::display()`,
  modifier bits, `Key` debug text, and command diagnostic id.
