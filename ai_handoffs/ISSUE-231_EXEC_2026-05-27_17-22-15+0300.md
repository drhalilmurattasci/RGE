## What I Changed

### Source

- `Invoke-AiDispatchQueue.ps1`:
  - **Worktree-isolation helpers** (`Resolve-DispatchWorktreePath`,
    `Test-DispatchWorktreeCleanupDecision`, `Format-DispatchWorktreeStatus`,
    `Format-DispatchWorktreeAuditSection`, `Format-DispatchOrphanRecoveryComment`):
    pure functions, no I/O, no git/gh/codex/claude calls. All five are
    dot-source-callable via the existing
    `RGE_AI_DISPATCH_QUEUE_SKIP_MAIN=1` testability seam.
  - **`Save-OrphanDispatchWorktree`**: now returns a `pscustomobject` with
    `Archived`, `ArchiveBranch`, `ArchivePath`, and `WorktreeMoved` fields
    (previously a bool). The `ArchivePath` field is the
    `.interrupt<N>` directory the orphan-recovery branch needs to surface
    to a human.
  - **`Invoke-OrphanRecovery`**: every comment-posting branch
    (`interrupted-publish` for ahead-of-origin, `already-published` for
    cleanup-only-interrupted, `interrupted` for genuine reset) now builds
    its GitHub comment via `Format-DispatchOrphanRecoveryComment`,
    feeding it the archive path produced by `Save-OrphanDispatchWorktree`
    so the comment names the `.interrupt<N>` path and emits the
    deterministic inspection / removal commands. The ahead-of-origin
    branch additionally archives a still-present worktree before
    commenting so the comment carries a path when one exists. Manual
    salvage review also fixed that ahead-of-origin path to report the
    archived branch name (`ai-dispatch/ISSUE-N.interruptN`) when the
    branch is renamed during archive, instead of pointing the human at
    the pre-archive branch name.
  - **`Write-DispatchLog`**: when `-WorktreeRoot` is supplied, embeds an
    `## Isolated Worktree` section in the committed audit log body via
    the new `Format-DispatchWorktreeAuditSection` helper. The section
    names the worktree path and the three deterministic commands a human
    needs (`git -C "<path>" status --short --branch`,
    `git -C "<path>" log --oneline -5`,
    `git worktree remove "<path>"`). The existing log location / git-scope
    plumbing through `-WorktreeRoot` is preserved.
  - **`Get-RepoRelativePathForQueue`**: now routes through
    `$script:DispatchWorktreeRoot` when set so all queue-side path
    formatting keys off the dispatch's repo root (the isolated worktree)
    rather than the primary checkout. After cleanup clears that variable
    the captured `$dispatchLogRel` is the single source of truth (pinned
    by `Issue231-DispatchLogPathStableAfterCleanup.Tests.ps1`).
  - **`Get-QueueStatusEntries`**: scopes its `git status --short` call
    through `git -C $script:DispatchWorktreeRoot` while a dispatch is
    in flight so the scope guard validates the worktree's content, not
    the primary's.
  - **Main flow** (the `try { ... } catch` block at the bottom):
    - Computes `$worktreePath` via `Resolve-DispatchWorktreePath` before
      any side effects, refuses to clobber a pre-existing worktree path
      (and tells the human how to inspect / remove it), then creates the
      worktree with `git worktree add -b $branch $worktreePath HEAD`.
      Manual CI follow-up changed `Resolve-DispatchWorktreePath` to use
      `[System.IO.Path]::Combine` instead of `Join-Path` so synthetic
      Windows drive paths in pure Pester tests do not require that drive
      to exist on GitHub-hosted runners.
    - Sets `$script:DispatchWorktreeRoot = $worktreePath` immediately
      after the add.
    - Runs `Invoke-AiDispatchLoop.ps1` under `Push-Location $worktreePath`
      so the inner loop's `git rev-parse --show-toplevel` resolves to the
      worktree, and the loop's run dir lands inside the worktree at
      `.ai/dispatch-<id>/`.
    - Routes the post-loop run dir read, `Write-DispatchLog`, scope
      guard, `git -C $worktreePath add -A`, and
      `git -C $worktreePath commit -F` through the worktree.
    - Captures `$dispatchLogRel = Get-RepoRelativePathForQueue $dispatchLogPath`
      immediately after `Write-DispatchLog` returns (while
      `$script:DispatchWorktreeRoot` is still set) and consumes that
      stable value at every later result-comment / PR-body / close-comment /
      commit-message site.
    - Preserves `main` / `branch` / `pr` / `-NoPublish` publish semantics
      verbatim from ISSUE-230. The only behavioral change is that the
      `main`-mode worktree-remove now precedes the `git branch -d`
      because git refuses to delete a branch that is checked out by a
      linked worktree; PR-mode removes the worktree after the PR opens
      and keeps the branch on `origin`; branch mode keeps both the
      branch and the worktree for human review; the no-commit empty
      branch deletes the worktree first.
    - On terminal state, delegates to
      `Test-DispatchWorktreeCleanupDecision` for the
      preserve/archive/remove decision; the result comment, stdout, and
      audit log all carry a `Format-DispatchWorktreeStatus` line naming
      the surviving path and the inspection / removal commands.
    - Retry-eligible failures archive both the branch and the worktree
      in lockstep under `.attempt<N>` (instead of just renaming the
      branch).
    - Removed the pre-ISSUE-231 stash / untracked-park dance entirely —
      with worktree isolation the primary's untracked files are outside
      the dispatch's working tree by construction.

## Open Questions for Reviewer

- The `interrupted-publish` orphan-recovery branch (ahead-of-origin)
   now archives a still-present worktree via `Save-OrphanDispatchWorktree`
   before posting its comment. The prior implementation did not archive
   the worktree on that path. Surfacing the archive path in the comment
   is the new ISSUE-231 reporting contract, and a still-present worktree
   on the ahead-of-origin path is an interrupted state, so archiving
   feels strictly in scope. Please confirm — if you prefer "leave the
   worktree untouched on this path", I will remove the archive call
   and the comment will simply not carry an `-ArchivePath`.
- The `Format-DispatchWorktreeAuditSection` integration test (in
   `Format-DispatchWorktreeAuditSection.Tests.ps1`) calls `git init`
   on a temp dir inside `BeforeAll`. That executes a real `git`
   process from within Pester. It does NOT touch the repo, network,
   gh / codex / claude, or any production state, but it is the only
   test in `tools/dispatch-tests/` that shells out. The alternative
   would be to stub `Git-Step`; the integration form is far more
   informative because it actually exercises the
   `Write-DispatchLog` -> `Format-DispatchWorktreeAuditSection`
   composition the queue depends on at runtime.
- The audit log carries the worktree path at write-time (BEFORE the
   disposition is known) because the audit log file is committed onto
   the dispatch branch BEFORE publish / cleanup decides preserve vs.
   archive vs. remove. The final disposition (preserved / archived /
   removed) is carried in the GitHub result comment, which is the
   durable post-disposition record. The audit-log section text
   explicitly defers the disposition narrative to the result comment;
   I believe this satisfies requirement #9's intent ("the audit log
   content includes the worktree path/status when preserved, archived,
   or removed"), but flagging for explicit reviewer attention since
   the prior CORRECT packet rejected an "audit log uses worktree-root
   for log location only" implementation.
