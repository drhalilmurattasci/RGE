## Acceptance Criteria

- For a Sweep profile with `n` vertices and path with `m` points, `SweepOp::brep_face_ids(owner)` returns exactly `2 + n * (m - 1)` pairs.
- Pair order exactly follows `SweepOp::evaluate`'s existing canonical face labels: `TopologyFaceId(0)` for first cap, `TopologyFaceId(1)` for last cap, then side faces in segment-major, profile-edge-major order using `TopologyFaceId(2 + segment_index * n + edge_index)`.
- `SweepFaceTag` uses frozen discriminants in that same order: first cap, last cap, side. Side identity must encode at least `segment_index`, `edge_index`, `profile_count`, and `path_segment_count` in deterministic little-endian bytes so profile-count and path-segment-count topology changes do not silently preserve side IDs.
- Cap identities are stable categorical cap identities for the owner and operator kind. They must not depend on Sweep numeric path coordinates.
- Repeated `brep_face_ids` calls for the same Sweep and owner return byte-identical IDs in byte-identical order.
- Numeric rebuilds that preserve `profile_count` and `path_segment_count` keep the same Sweep B-Rep face IDs; tests must cover this for caps and sides.
- Sweep side IDs are unique within one owner, disjoint across different owners, and disjoint from at least one existing non-Sweep face-id namespace via the operator-kind separator.
- Direct `brep_face_ids_for_node` resolution for a Sweep node returns the same ordered IDs as the direct provider if the resolver is edited.
- Existing Sweep label tests remain valid; do not rewrite the `SweepOp::evaluate` face-label contract to make the provider easier.
- The new or revised tests are rustfmt-clean. The prior attempt failed only at `cargo +nightly fmt --check`; this dispatch is not acceptable unless formatting is clean.
