## Deliverables

- Camera pan math that translates `EditorCameraState.eye` and `EditorCameraState.target` by the same finite view-plane vector while preserving the eye-target offset, FOV, clip planes, up vector, and finite state.
- Lifecycle-private middle-button pan drag state/routing, preferably alongside the existing right-button orbit helper in `lifecycle/viewport_navigation.rs`.
- `WindowEvent` routing that starts pan only on middle-button press while the cursor is over the Viewport tab body, updates pan on cursor movement while active, and stops pan on middle-button release even when egui consumed the release.
- Focused `rge-editor-shell` tests in `crates/editor-shell/src/lifecycle/tests.rs` covering pan start gating, cursor-delta math, release stop behavior, no-cursor/no-host/non-viewport no-ops, non-finite/zero/degenerate no-ops, and preservation of left face-pick, wheel zoom, and right orbit behavior.
- Task/status doc updates limited to `.ai/dispatch.tasks.md` and `Status.md`, marking task 119/ISSUE-373 consistently with repository convention if implementation completes.

## Acceptance Criteria

- Pressing middle mouse while the pointer is over the current Viewport tab body starts a shell-private pan drag; moving the cursor while that drag is active translates both `EditorShell::editor_camera.eye` and `editor_camera.target` in the camera view plane.
- The pan preserves `eye - target`, `up`, `fov_y_radians`, `near`, and `far`; it must not mutate CAD graph/projection state, selection, undo/dirty state, menus, commands, accelerators, palette state, or Cargo metadata.
- Zero cursor delta, missing cursor position, missing egui host, non-viewport press, non-finite cursor positions, non-finite deltas, non-finite camera vectors, and degenerate eye-target/up vectors are no-ops. They must not panic and must not leave partial state behind.
- Middle-button release stops pan even if egui consumed that release. Release should be unconditional for an active middle-button pan, matching the existing right-button orbit release discipline.
- Left-button face-pick routing remains governed by `should_fire_face_pick`; mouse-wheel zoom remains viewport-only; right-button orbit still starts/stops/updates as before; View reset/zoom commands, menu/palette routing, and keyboard accelerator routing are unchanged.
- No `Command` enum, menu registry entry, accelerator, command-palette behavior, host-shell FIFO behavior, generalized command registry, plugin runtime/discovery/loading, OS clipboard behavior, CAD mutation path, undo/dirty policy, Cargo manifest, or `Cargo.lock` change is introduced.
