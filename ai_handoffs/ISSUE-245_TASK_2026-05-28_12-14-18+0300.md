## Goal

Wire the existing W04 raw Wasmtime Cranelift cold-start focused release test
(`crates/script-bench/src/wasmtime_cranelift.rs::tests::w04_cold_start_wasmtime_cranelift`)
into GitHub Actions as an observational signal. The CI job runs the test on the
standard `ubuntu-latest` runner, captures the `duration_ns` / `duration_ms`
stdout, and emits the value to the Actions job summary
(`$GITHUB_STEP_SUMMARY`).

The job must not add timing-based gating: no new threshold, no comparison
against the DENTAL recorder-host baseline (255,600 ns / 0.256 ms), and no new
bench harness. The harness's existing `< 2s` ceiling assertion is preserved
unchanged and is the only timing-related failure path; the 2-second ceiling is
structurally many orders of magnitude above any plausible CI sample and is
expected never to trip. A compile error or test panic still fails the job
(cargo-test default behavior), which is intentional: a broken test is a real
failure, distinct from slow timing.

This is a small Phase 9 CI-wiring dispatch. It must reuse the existing focused
release test. It must not change Rust source, tests, benches, Cargo metadata,
architecture plans, dispatch automation, or any other workflow.

## Deliverables

- A new job in `.github/workflows/bench.yml`, proposed id and check name
  `script-bench-cold-start-observational`, alongside the existing
  `script-bench-compile` job.
- The new job runs on `ubuntu-latest` and uses the same workflow triggers as
  the existing bench workflow (`push` to `main`, `pull_request` to `main`).
- The new job installs the pinned toolchain via the same
  `dtolnay/rust-toolchain@stable` plus `Swatinem/rust-cache@v2` pattern used
  by the existing bench job.
- The new job invokes the existing focused release test three times in
  succession:

```bash
cargo test -p rge-script-bench --release --lib wasmtime_cranelift::tests::w04_cold_start_wasmtime_cranelift -- --nocapture
```

- The job captures each run's `duration_ns` / `duration_ms` stdout line and
  computes the minimum of three samples in bash arithmetic. Do not add a helper
  script, Python, a bench harness, or a threshold environment variable.
- The job emits a Markdown table to `$GITHUB_STEP_SUMMARY` containing the three
  per-run `duration_ns` values, the min-of-3 `duration_ns`, the toolchain
  version, the wasmtime version read from `Cargo.lock`, and the caveat block
  required by the acceptance criteria.
- The job uses default cargo-test failure semantics (`continue-on-error` unset
  or `false`). The job fails on compile error, test panic, or the harness's
  existing `< 2s` ceiling assertion. The job adds no new timing assertion, no
  threshold environment variable, and no comparison-against-baseline logic.
- Concise forward-only updates to `Status.md`, `HANDOFF.md`, and `change.md`
  noting that ISSUE-245 wired W04 cold-start into CI as an observational signal.
  Do not rewrite dated historical snapshots.

## Planner Notes

This packet follows the orchestrator semantics in `AI_DISPATCH_AUTOMATION.md`:
TASK plus EXEC packets, plus CORRECT only if a correction round is needed.
Missing REVIEW or CLOSEOUT packets are normal for orchestrator dispatches.

The intended publication path is queue PR mode
(`Invoke-AiDispatchQueue.ps1 -PublishMode pr`), so the dispatch branch is pushed
and a PR is opened for human review without merging to `origin/main` and without
closing the GitHub issue automatically.

The observational shape is deliberate. The DENTAL recorder host
(Ryzen 7 7800X3D / RTX 4060 Ti / Windows 11 Pro for Workstations) is the
canonical numbers source for W04 cold-start; GitHub-hosted `ubuntu-latest`
runners are a different runner class with noisier and structurally different
timings. A CI-host-specific baseline row is a separate follow-up dispatch, not
this one. Self-hosted DENTAL runners are a later follow-up.

If the CI sample emerges as useful (visible, low-flake, occasionally
interesting), candidate follow-up chips include:

- CI-host-specific baseline row in `BASELINE.md`.
- PR-comment posting step that prints the min-of-3 inline on the PR.
- Self-hosted DENTAL runner registration.
- Trend file in-repo, such as `crates/script-bench/CI_HISTORY.md`, that appends
  a row per main merge.

None of those are this dispatch. Each is a separate chip that must preserve the
no-gate semantics established here.
