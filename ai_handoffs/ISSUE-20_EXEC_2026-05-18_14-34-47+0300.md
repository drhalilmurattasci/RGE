### Tests

- `crates/cad-core/tests/phase_7_2_gate_closure.rs` — extended the
  Phase 7.2 rebuild gate to include `SweepOp`:
  - Imports: added `Polyline3D` and `SweepOp` from `rge_cad_core`.
  - `RootKind::Sweep` added as a fifth root kind.
  - `RootSeed::Sweep { profile_n, profile_radius, path_m, path_z_step,
    path_drift }` variant added — `profile_n` and `path_m` are the two
    topology counters; `profile_radius`, `path_z_step`, `path_drift` are
    numeric (topology-preserving) coordinates.
  - `generate_random_chain` now draws `next_range_u32(5)` and maps the
    fifth bucket to `RootKind::Sweep` (Cuboid/Extrude/Revolve/Loft moved
    to explicit `0..=3` arms; behavior otherwise unchanged).
  - `generate_root_seed` builds a Sweep seed: `profile_n` in `3..=6`,
    `path_m` in `2..=5` (1..=4 segments), `path_z_step` in `0.5..=3.0`
    (strictly positive → monotonic-Z), `path_drift` per-component in
    `-1.0..=1.0`.
  - New local helper `make_monotonic_z_path(m, z_step, drift)` builds a
    `Polyline3D` whose point `k` is `(k*drift_x, k*drift_y, k*z_step)`;
    positive `z_step` guarantees strict monotonic Z and
    adjacent-point distinctness, so the path is always valid Sweep
    input. The Sweep profile reuses the existing `make_regular_polygon`
    (a strictly convex regular polygon).
  - `add_root` constructs `OperatorNode::Sweep(SweepOp::new(profile,
    path))` through public `rge_cad_core` APIs only.
  - `generate_random_mutation`: `root_supports_topology_change` now
    includes `RootKind::Sweep`.
  - `apply_preserving_mutation` Sweep arm: 3-way pick over
    `profile_radius` / `path_z_step` / `path_drift`, all keeping
    `profile_n` and `path_m` fixed (profile vertex count and path
    segment count `m-1` unchanged).
  - `apply_changing_mutation` Sweep arm: coin flip between
    `pick_different_profile_n` and the new `pick_different_path_m`
    (picks a different point count in `2..=5`); both keep the Sweep
    valid and change a topology counter.
  - New local helper `pick_different_path_m`.
- `crates/cad-core/tests/phase_7_2_gate_closure.rs::phase_7_2_gate_closure_100_chains_10_rebuilds_seed_0x7e5a_dead_beef_c0de`

  `profile_n` change or a `path_m` (path point count) change. Both
  always change the face+edge ID vector *length* (face count
  `2 + n*(m-1)`, edge count `n*(2s+1)`), so the gate's `assert_ne!`

  different-IDs Sweep changing case (no Sweep analogue to Revolve's

  realization of "vector-level inequality" for Sweep — it matches the
  existing Extrude/Loft `profile_n`-change behavior.
