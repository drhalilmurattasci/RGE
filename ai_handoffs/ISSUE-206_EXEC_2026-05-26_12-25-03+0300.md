## Protocol Note — Orchestrator Dispatches Do Not Produce REVIEW or CLOSEOUT Packets

`AGENTS.md` (loaded by `CLAUDE.md`) is explicit: an orchestrator dispatch
driven by `Invoke-AiDispatchLoop.ps1` writes only a `TASK`, an `EXEC`, and
optionally a `CORRECT` packet to `ai_handoffs/`. The Codex control review
lives in the gitignored run-dir JSON
(`.ai/dispatch-<DispatchId>/codex.control.round<N>.json`), never as a
`_REVIEW_` packet. No `_CLOSEOUT_` packet is emitted, ever. Completion is
signalled by the inner-loop `Dispatch loop finished.` message together with
a Codex `pass` verdict and the queue-run commit on the dispatch's branch —
*not* by a `_CLOSEOUT_` artifact.

Consequence for ISSUE-202 and ISSUE-204: the *absence* of `_REVIEW_` or
`_CLOSEOUT_` packets in `ai_handoffs/` for those dispatches is not evidence
of unfinished work. Both dispatches are by-design complete after their
final EXEC packet + Codex `pass` verdict + queue-run commit landed on
their respective `ai-dispatch/ISSUE-<n>` branches.

## Section 1 — `PublishMode branch` → `-NoPublish` Propagation in `Invoke-AiDispatchAuto.ps1`

The Auto driver translates `PublishMode` into queue arguments in the
"Drain: run the hardened queue on the pending issue" block beginning around
line 889. The propagation is three lines (L895-899) inside that block:

```powershell
# Invoke-AiDispatchAuto.ps1 L895-899
$queueArgs = @('-NoProfile', '-ExecutionPolicy', 'Bypass', '-File', $queueScript,
    '-MaxPlanRevisions', $MaxPlanRevisions, '-MaxCorrectionRounds', $MaxCorrectionRounds)
if ($PublishMode -eq 'branch') { $queueArgs += '-NoPublish' }
if ($TraceTiming) { $queueArgs += '-TraceTiming' }
if ($EnablePreflightAudit) { $queueArgs += '-EnablePreflightAudit' }
```

Key points:

- `$queueArgs` is the array splatted into `& powershell.exe @queueArgs` at
  L905 to launch the queue runner as a child process.
- L897 — `if ($PublishMode -eq 'branch') { $queueArgs += '-NoPublish' }` —
  is the single site that translates the documented branch mode into the
  Queue's `-NoPublish` switch. In `main` mode this append never executes,
  so `-NoPublish` is not passed and Queue defaults to its auto-publish path.
- L898 — `if ($TraceTiming) { $queueArgs += '-TraceTiming' }` — is an
  **independent** conditional append controlled by the
  `[switch]$TraceTiming` parameter declared at L79 in the Auto `param()`
  block, with an environment-variable fallback at L86
  (`$env:RGE_AI_DISPATCH_TRACE_TIMING -match '^(1|true|yes|on)$'`). Note
  that the L898 conditional is gated on the parameter only, not the
  env-var fallback, so the env-var fallback only enables Auto's own
  in-process tracing and does not implicitly forward `-TraceTiming` to
  Queue. Either way, `-TraceTiming` controls only the timing-trace JSONL
  emission and has no relationship to publish policy.
- L899 — `if ($EnablePreflightAudit) { $queueArgs += '-EnablePreflightAudit' }`
  — is the third independent conditional, wired in by ISSUE-204. Like
  `-TraceTiming` it is orthogonal to publish policy.

Header context is consistent with the implementation: the script's
comment-based help (read inline because the running session blocks
`Get-Help` from cold-running the script under sandbox) declares
`-PublishMode` at L34-35 with allowed values `branch` / `main` and the
default `'branch'`, and `[ValidateSet('branch', 'main')]` /
`[string]$PublishMode = 'branch'` at L63-64.

Conclusion for Auto: the script does exactly what its help text says — in
`branch` (default) mode it appends `-NoPublish` to the queue invocation,
and in `main` mode it does not.

## Section 2 — `NoPublish` Receipt and Consumption in `Invoke-AiDispatchQueue.ps1`

The Queue script accepts `-NoPublish` as a parameter and consumes it in the
publish block at the end of a queue tick.

Parameter declaration (L43-64, `param()` block):

```powershell
# Invoke-AiDispatchQueue.ps1 L53
[switch]$NoPublish,
```

The script's comment-based help mentions `-NoPublish` at L41 ("Pushes only
successful, Codex-control-passed dispatch commits. Use -NoPublish to keep
the old local-branch-only behavior for a one-off run.").

Publish-eligibility computation and consumption (L1360-1430, "Publish
passed work" block):

```powershell
# Invoke-AiDispatchQueue.ps1 L1360
$eligibleForPublish = ($committed -and $loopExit -eq 0 -and $verdict -eq 'pass')

if ($eligibleForPublish -and -not $NoPublish) {
    Write-Output "Codex control passed; publishing $branch to origin/main."
    Write-TimingTrace "queue.publish: block-entry; eligibleForPublish=true"
    # ... git fetch, ff-merge, push origin main, branch-delete ...
} elseif ($eligibleForPublish -and $NoPublish) {
    $publishDetail = "NoPublish set; kept $branch local."
    Write-TimingTrace "queue.publish: skipped (NoPublish=true, eligibleForPublish=true)"
} elseif ($committed) {
    $publishFailed = $true
    $publishDetail = "Not published because loop exit code was $loopExit and Codex control verdict was '$verdict'."
    Write-TimingTrace "queue.publish: skipped (eligibleForPublish=false, loopExit=$loopExit, verdict=$verdict)"
} else {
    $publishDetail = "No branch commit was created."
    Write-TimingTrace "queue.publish: skipped (committed=false)"
}
```

Key points:

- `$eligibleForPublish` (L1360) is computed *independently of* `$NoPublish`.
  Its three inputs are `$committed` (the per-issue branch commit was
  written), `$loopExit -eq 0` (the inner dispatch loop exited 0), and
  `$verdict -eq 'pass'` (Codex control returned `pass`). This is the
  success gate documented in `AI_DISPATCH_AUTOMATION.md` §2 and §6 — "the
  queue runner publishes only when the loop exits 0 and Codex control
  verdict is `pass`."
- The first `if` branch (L1362) — `if ($eligibleForPublish -and -not
  $NoPublish)` — is the only path that runs the actual `git fetch` /
  ff-merge / `git push origin main` / branch-delete sequence.
- The second `elseif` branch (L1420-1422) — `elseif ($eligibleForPublish
  -and $NoPublish)` — is the **publish-skip branch**: the run was eligible
  by every other criterion, but `-NoPublish` was set, so the push is
  deliberately skipped, the local branch is kept, and the trace line
  `queue.publish: skipped (NoPublish=true, eligibleForPublish=true)` is
  emitted. Note that by this point the per-issue branch commit has already
  been written (see L1438 — "Committed locally as `<sha>` on branch
  `<branch>` (NoPublish set; not pushed)."). Skipping publish therefore
  preserves the work on the dispatch branch for human review and merge.

Independence from `-TraceTiming`: the Queue's `-TraceTiming` switch (L61)
controls only the JSONL trace emission via `Write-TimingTrace` (L92-148).
`Write-TimingTrace` no-ops when tracing is disabled and never affects
control flow. The publish-eligibility computation and the publish-skip
branch above do not read `-TraceTiming` at all; they read only
`$committed`, `$loopExit`, `$verdict`, and `$NoPublish`. The trace line
`queue.publish: skipped (NoPublish=true, eligibleForPublish=true)` is
emitted from inside the skip branch regardless of why the branch was
taken — `-TraceTiming` only decides whether the line is materialized to
the JSONL file. In short, `-TraceTiming` *exposes* the publish decision;
it does not *make* the decision.
