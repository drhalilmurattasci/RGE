## Goal

Add a Codex-only stall watchdog to `Invoke-WithTimeout` for GitHub issue #182.
The dispatch loop already has a hard timeout for native commands, but a Codex
process can remain alive after producing partial log output and then stop
making progress. This dispatch must detect that stalled Codex log state, kill
the process tree, and report it distinctly without changing legacy timeout
behavior or any Claude, verification, queue, auto, scheduler, Rust, Cargo, doc,
or schema surface.

## Deliverables

- Extend `Invoke-WithTimeout` with optional integer parameters
  `StallThresholdSec` and `PollIntervalSec`.
- Preserve exact legacy behavior when `StallThresholdSec` is omitted or `0`:
  no stall polling runs, the existing hard-timeout path remains in effect, the
  old `Code=124` timeout result remains unchanged, and existing callers that do
  not opt in keep the same behavior.
- When `StallThresholdSec` is greater than `0`, poll the child process until it
  exits, hits the existing hard timeout, or hits the new stall threshold.
- Arm the stall watchdog only after `OutFile` exists and has non-zero size.
  Time before the first non-empty `OutFile` must not count toward the stall
  threshold.
- After the watchdog is armed, treat output activity as an increase in
  `OutFile` length or an advance in `OutFile.LastWriteTimeUtc`. Do not refresh
  the stall timer merely because a non-empty file still exists.
- If the stall threshold is exceeded while the process is still running, kill
  the same process tree with `taskkill /T /F /PID`, return `Code=125`,
  `TimedOut=$true`, and `Stalled=$true`, and do not continue waiting for the
  hard timeout.
- If the hard timeout is reached before any stall condition, preserve the
  existing hard-timeout semantics: kill the process tree, return `Code=124`,
  `TimedOut=$true`, and, when the stall path was enabled, `Stalled=$false`.
- Add a top-level dispatch-loop parameter named `CodexStallThresholdSec` with a
  default of `300` seconds. A value of `0` must disable the Codex stall
  watchdog.
- Wire the watchdog only through `Invoke-CodexPrompt` by passing
  `-StallThresholdSec $CodexStallThresholdSec` to its `Invoke-WithTimeout`
  call. Do not pass the stall parameters from Claude calls, Claude readiness
  probes, verification calls, git calls, or any other call site.
- Update `Invoke-CodexPrompt` timeout handling so a stalled Codex process fails
  with a stall-specific message that names the threshold and log path, while
  non-stall hard timeouts and non-zero exits keep their existing behavior.

## Acceptance Criteria

- `Invoke-WithTimeout -StallThresholdSec 0` and existing calls that omit

  verification, and any other non-Codex callers have unchanged process launch,
  stream redirection, hard-timeout, exit-code, and cleanup behavior.
- `PollIntervalSec` cannot produce a tight polling loop; use a positive default
  and PowerShell 5.1-compatible validation or clamping.
- The watchdog never starts counting stall time until `OutFile` has non-zero
  size.

  `TimedOut=$true`, and `Code=125`.

  `Code=124`.
- `Invoke-CodexPrompt` is the only call site that opts into stall detection, and

## Planner Notes

The existing `Invoke-WithTimeout` wraps the native command in a child
`powershell.exe`, writes stdout/stderr to files, and uses `taskkill /T /F /PID`
for hard timeouts. Keep that process-tree kill model for the stall watchdog so
the failure mode is consistent with the existing infrastructure timeout.

Use PowerShell 5.1-compatible polling primitives such as `Start-Sleep`,
`Get-Item`, `DateTime.UtcNow`, and `LastWriteTimeUtc`. When both hard-timeout
and stall-watchdog logic are active, cap each sleep by the nearest pending
deadline so the process is killed promptly and the function does not overshoot
the configured hard timeout by a full polling interval.

The result code distinction is intentional: `124` remains the legacy hard
timeout code, while `125` identifies a Codex stall. `TimedOut=$true` is still
required for stalls because existing caller logic treats infrastructure
timeouts as terminal failures, and `Stalled=$true` gives `Invoke-CodexPrompt`
the extra context needed for a clearer error message.

Do not use this dispatch to improve prompts, add checklists, audit stale
worktree state, or change control-review expectations. Those are explicit
non-goals for ISSUE-182.
