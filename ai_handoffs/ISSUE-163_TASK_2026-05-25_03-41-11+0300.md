## 5-Question Typed ComponentValue Bridge Preflight Answer Block

### Q1. Bridge owner plus dependency direction

Answer: No existing crate currently owns the full typed `rge_data::Scene` to
`kernel/ecs::World` bridge. The current dependency evidence narrows the valid
direction to a caller above `rge-data`, `kernel/ecs`, and the component crates,
but it does not name an existing bridge owner.

Current-code anchors: `rge-data` is documented as a stateless serde plus
migration layer (`crates/rge-data/src/lib.rs:13`) and exports the schema
envelope types only (`crates/rge-data/src/lib.rs:67`). Its scene module says
component payloads are reflection-neutral and that downstream callers later
walk `ComponentValue::data` through `kernel/types::serde_bridge::from_ron`
keyed by `ComponentValue::type_id` (`crates/rge-data/src/scene.rs:21`,
`crates/rge-data/src/scene.rs:90-94`). `rge-data` depends on
`rge-kernel-asset`, `serde`, `ron`, `ulid`, and `thiserror`, and has only a
test-only `rge-kernel-ecs` dev-dependency (`crates/rge-data/Cargo.toml:12-20`);
it does not depend on the component crates or `kernel/types`.

Plausible candidates and dependency implications:

- `crates/rge-data`: would need `rge-data -> rge-components-spatial`,
  `rge-data -> rge-components-render`, `rge-data -> rge-components-visibility`,
  `rge-data -> rge-kernel-ecs`, and likely `rge-data -> rge-kernel-types`.
  That would violate the crate's documented schema-only posture even if the
  forbidden-dep lint does not specifically ban Tier-2 to Tier-2 here.
- `kernel/types` or `kernel/ecs`: cannot own the bridge because Tier 1 kernel
  crates cannot depend on Tier 2 crates under forbidden-dep rule 1, and the
  bridge would need `rge-data` plus component crates. These crates can provide
  identity, reflection, and ECS insertion primitives, not game-domain mapping.
- A new `kernel/scene-bridge` cavity: same Tier 1 problem as other kernel
  crates if it depends on `rge-data` or component crates. This would be a new
  substrate decision, not current code.
- A new Tier-2 `rge-scene-loader` or equivalent scene-loader crate: could
  depend downward/across on `rge-data`, `rge-kernel-ecs`, `rge-kernel-types`,
  and the component crates while preserving `rge-data` as schema-only and
  avoiding renderer-tier ownership. This is the most plausible dependency
  direction, but the crate does not exist today.
- `runtime/runtime-*` binaries: currently have empty dependency sections
  (`runtime/runtime-headless/Cargo.toml:16`, `runtime/runtime-desktop/Cargo.toml:16`).
  They could depend on the needed crates without tripping rule 6, but this
  would duplicate bridge logic per runtime binary unless a reusable owner is
  introduced.
- `crates/editor-shell` or `editor/rge-editor`: editor crates may depend on
  game-domain and renderer crates because forbidden-dep rule 6 targets
  renderer crates, not editor crates. `editor-shell` already wraps a
  `rge_kernel_ecs::World` (`crates/editor-shell/src/world.rs:47-57`) and
  exposes `kernel_mut()` for typed insertion
  (`crates/editor-shell/src/world.rs:100-102`), but it does not currently
  depend on `rge-data` or the scene component crates. This would be an
  editor-specific bridge owner, not a general runtime bridge.
- Component crates: `rge-components-spatial`, `rge-components-render`, and
  `rge-components-visibility` currently depend only on `serde` plus their
  local needs (`crates/components-spatial/Cargo.toml:12-16`,
  `crates/components-render/Cargo.toml:12-17`,
  `crates/components-visibility/Cargo.toml:12-16`). They can own impls for
  their local types, but no single component crate can own a whole scene bridge
  spanning all component families without importing `rge-data` and unrelated
  component crates.
- Renderer crates `rge-gfx`, `rge-gfx-ir`, and `rge-brep-render`: forbidden-dep
  rule 6 says renderer crates cannot depend on game-domain crates
  (`tools/architecture-lints/src/forbidden_dep.rs:38-46`,
  `tools/architecture-lints/src/forbidden_dep.rs:173-179`), and
  `rge-components-*` are explicitly game-domain
  (`tools/architecture-lints/src/forbidden_dep.rs:46`,
  `tools/architecture-lints/src/forbidden_dep.rs:446`). Renderer ownership is
  therefore forbidden.
- `crates/script-host`: it has a hard-coded `Counter` ECS bridge and explicitly
  defers the generic component bridge (`crates/script-host/src/ecs_bridge.rs:5-8`,
  `crates/script-host/src/ecs_bridge.rs:33-44`). It is not a scene loader owner.
- A test-local helper: current `rge-data` load+tick testing has an
  identity-only helper that spawns ECS entities but deliberately ignores
  component payloads
  (`crates/rge-data/tests/golden_simple_scene_load_tick.rs:11-17`,
  `crates/rge-data/tests/golden_simple_scene_load_tick.rs:37-42`). This can
  validate tests, not production architecture.

Conclusion: existing dependency rules justify several "must not live here"
answers and suggest that any production bridge must sit above the schema,
kernel, and component crates. They do not justify a concrete existing owner.

### Q2. type_id to component-type mapping mechanism

Answer: No current production mapping mechanism exists for
`ComponentValue.type_id` string to concrete Rust component type.

`ComponentValue` stores `type_id: String` and `data: String`
(`crates/rge-data/src/scene.rs:90-94`). `kernel/types::TypeId` is a 128-bit
hash generated from a fully qualified type name
(`kernel/types/src/type_id.rs:55-64`), while `Reflect` exposes
`TYPE_ID`, `TYPE_NAME`, `FQ_TYPE_NAME`, schema version, field descriptors, and
dynamic field access (`kernel/types/src/reflect.rs:68-84`). The
`serde_bridge` helpers are generic functions, not keyed dispatch:
`to_ron<T: Reflect + Serialize>` and
`from_ron<T: Reflect + DeserializeOwned>` require the concrete `T` at the call
site (`kernel/types/src/serde_bridge.rs:117-132`).

The reflect crate also documents no global runtime registry
(`kernel/types/src/lib.rs:40`). The derive macro matches that: it emits one
`impl Reflect` and no registry or inventory submission
(`crates/macros-reflect/src/codegen.rs:1-6`). It builds `FQ_TYPE_NAME` from
`module_path!()` plus the Rust type name (`crates/macros-reflect/src/codegen.rs:97`,
`crates/macros-reflect/src/codegen.rs:112`), but it currently emits
`TYPE_ID` as all zero bytes (`crates/macros-reflect/src/codegen.rs:110`). It
also rejects enums and non-named structs today
(`crates/macros-reflect/src/derive.rs:35-43`).

The four simple-scene component types do not derive `Reflect`:

- `Transform` derives `Debug`, `Clone`, `Copy`, `PartialEq`, `Serialize`, and
  `Deserialize` (`crates/components-spatial/src/transform.rs:22-23`).
- `Camera` derives `Debug`, `Clone`, `Copy`, `PartialEq`, `Serialize`, and
  `Deserialize`; its nested `Projection` enum has the same derive surface
  (`crates/components-render/src/camera.rs:8-9`,
  `crates/components-render/src/camera.rs:43-44`).
- `Light` derives `Debug`, `Clone`, `Copy`, `PartialEq`, `Serialize`, and
  `Deserialize`; its nested `LightKind` enum has the same derive surface
  (`crates/components-render/src/light.rs:18-19`,
  `crates/components-render/src/light.rs:56-57`).
- `Visibility` derives `Debug`, `Default`, `Clone`, `Copy`, `PartialEq`, `Eq`,
  `Hash`, `Serialize`, and `Deserialize`
  (`crates/components-visibility/src/visibility.rs:19-20`).

The broader public component-crate sweep reaches the same shape: spatial
exports `ChildOf`, `Entity`, `GlobalTransform`, `Parent`, and `Transform`
(`crates/components-spatial/src/lib.rs:32-36`), render exports `AssetId`,
`NULL_ASSET_ID`, `BRepHandle`, `CadNodeId`, `Camera`, `Projection`, `Light`,
`LightKind`, `Lod`, `LodLevel`, `MaterialHandle`, `MeshHandle`,
`ReflectionProbe`, and `SkinnedMesh`
(`crates/components-render/src/lib.rs:31-40`), and visibility exports
`Disabled`, `Hidden`, `Highlight`, and `Visibility`
(`crates/components-visibility/src/lib.rs:25-28`). The concrete public structs
and enums in those crates derive serde-oriented traits such as
`Serialize`/`Deserialize`; the inspected current derive lines do not show
`Reflect`, `Component`, or `SnapshotComponent` derives or impls
(`crates/components-spatial/src/child_of.rs:14-15`,
`crates/components-spatial/src/entity.rs:14-16`,
`crates/components-spatial/src/global_transform.rs:17-18`,
`crates/components-spatial/src/parent.rs:17-19`,
`crates/components-render/src/brep_handle.rs:15-16`,
`crates/components-render/src/lod.rs:12-37`,
`crates/components-render/src/mesh_handle.rs:15-17`,
`crates/components-render/src/reflection_probe.rs:13-14`,
`crates/components-visibility/src/disabled.rs:11-12`,
`crates/components-visibility/src/hidden.rs:10-11`,
`crates/components-visibility/src/highlight.rs:18-19`).

The `rge-data` schema test now pins string constants for
`rge::components::Transform`, `Camera`, `Light`, and `Visibility`, but that is
test evidence for fixture shape, not a production registry or dispatch table
(`crates/rge-data/tests/golden_simple_scene_schema.rs:24-34`). Current code
therefore has canonical strings in schema tests, but no current mechanism that
maps those strings to Rust types and invokes `from_ron` for the correct type.

### Q3. ECS insertion target

Answer: The current ECS target for entity identity is clear, but typed
component attachment for the four scene component types is not usable as-is.

`rge-data::EntityId` is a transparent ULID wrapper
(`crates/rge-data/src/entity_ref.rs:37-38`), and `kernel/ecs::EntityId` also
wraps a ULID and exposes `EntityId::from_ulid`
(`kernel/ecs/src/entity.rs:22`, `kernel/ecs/src/entity.rs:37-43`). The current
load+tick helper converts `rge_data` entity IDs to `kernel/ecs` IDs and calls
`World::spawn_with_id` (`crates/rge-data/tests/golden_simple_scene_load_tick.rs:37-42`).
The production `World` surface supports that entity creation directly
(`kernel/ecs/src/world.rs:52`, `kernel/ecs/src/world.rs:133-156`).

For component attachment, `kernel/ecs::World` exposes typed insertion through
`World::insert<C: Component>` (`kernel/ecs/src/world.rs:246-253`),
`World::spawn_with<C: Component>` (`kernel/ecs/src/world.rs:178-183`),
`EntityMut::insert<C: Component>` (`kernel/ecs/src/entity.rs:164-166`), and
free-function aliases such as `insert_component<C: Component>`
(`kernel/ecs/src/lib.rs:58-84`). The erased insertion path exists only as
`pub(crate) fn insert_erased` for snapshot restore
(`kernel/ecs/src/world.rs:160-175`,
`kernel/ecs/src/snapshot.rs:350`) and is not a public arbitrary scene-loader
API.

The ECS `Component` trait is a manual marker trait requiring
`Any + Send + Sync + 'static`; the docs say there is no derive macro and
callers must implement it manually (`kernel/ecs/src/component.rs:11-20`).
`SnapshotComponent` additionally requires `Component + Serialize +
DeserializeOwned` and opt-in registration
(`kernel/ecs/src/snapshot.rs:51-66`,
`kernel/ecs/src/snapshot.rs:165-172`).

The four scene component types are serializable, but they do not currently
implement `rge_kernel_ecs::Component` or `SnapshotComponent`. A third bridge
crate cannot implement the external `Component` trait for external component
types because of Rust's orphan rules; those impls would have to live in the
owning component crates or use new wrapper types. Either path is new source
design outside this read-only audit.

### Q4. Justified-from-current-code check

Answer: NO.

The bridge owner plus dependency direction is only partially constrained by
current code: `rge-data`, kernel crates, renderer crates, and script-host are
not viable owners, while a higher Tier-2 loader, runtime owner, or editor owner
is plausible. But no existing crate currently owns the full bridge as-is.

The type-id mapping mechanism is not justified by current code. The repo has
schema-level strings, `TypeId`, `Reflect`, generic `serde_bridge` helpers, and
macro-emitted `FQ_TYPE_NAME`, but it has no registry, match table, inventory,
or existing `Reflect` derive on the four component types that maps
`ComponentValue.type_id` to a concrete Rust type. The ECS insertion step also
requires the target types to implement `Component`, which they do not today.

Because both the owner decision and the type-id mapping mechanism are not
current-code surfaces, Q4 cannot be `YES`.

### Q5. Smallest implementation dispatch or NEEDS_HUMAN

Answer: NEEDS_HUMAN.

Required arbiter decisions:

1. Choose the bridge owner and dependency direction: new Tier-2 scene-loader
   crate, runtime-owned bridge, editor-owned bridge, or another explicit owner.
2. Choose the type-id mapping mechanism: explicit registry or match table,
   `Reflect`-based mapping, schema-string convention, or another approved
   design.
3. Choose how scene component types become ECS components: add `Component`
   impls in the owning component crates, use wrapper types, or choose a
   different attachment strategy.

No implementation dispatch is safe before those decisions, because current
code does not provide the bridge owner or mapping mechanism as-is and the four
typed component types would need a new trait impl or wrapper surface to be
attached to `kernel/ecs::World`.

Recommendation: NEEDS_HUMAN
