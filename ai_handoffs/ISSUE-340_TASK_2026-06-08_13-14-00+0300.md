## Goal

Add an explicit, operator-controlled Codex execution option for build-measurement dispatches that need fresh external scratch targets under `B:\sdk`, while preserving the current workspace-sandbox default for normal dispatches. This is automation-substrate work motivated by ISSUE-338 and ISSUE-339, where the queued Codex executor could not create the required `B:\sdk` targets and the measurements had to be completed by manual salvage.

## Deliverables

- A Codex-only `-CodexExecutorExternalScratch` switch, or an equally explicit and narrowly named equivalent, on `Invoke-AiDispatchLoop.ps1`.
- Pass-through for that option through `Invoke-AiDispatchQueue.ps1`, `Invoke-AiDispatchAuto.ps1`, `Invoke-AiDispatchGuard.ps1`, and `Register-AiDispatchSchedule.ps1` where those entry points can launch Codex execution.
- Loop behavior that keeps the default Codex execution sandbox at `workspace-write`, and changes only the Codex execution phase to the external-scratch-capable sandbox when the option is supplied.
- Focused `tools/dispatch-tests` coverage proving default workspace scope, opt-in external-scratch invocation, pass-through argument construction, and unchanged publish-mode behavior.
- Documentation/status updates recording the operator intent, safety boundary, ISSUE-338/ISSUE-339 provenance, and the exact future command for `B:\sdk` measurement dispatches.
- One EXEC packet reporting the implementation, changed files, verification commands, and any deviations.

## Acceptance Criteria

- No-flag dispatch behavior is unchanged: Codex execution still invokes `codex exec` with `--sandbox workspace-write`.
- Opt-in behavior is deliberate: `-CodexExecutorExternalScratch` changes only Codex execution to the external-scratch-capable sandbox, expected to be `danger-full-access` unless implementation discovers a narrower Codex CLI mode that genuinely permits `B:\sdk`.
- Plan fill, Codex plan gate, preflight audit, correction packet authoring, and Codex control review retain their existing sandbox routing; do not broaden planner or reviewer phases.
- The option is Codex-only. If it is supplied with `-Executor claude`, fail fast with a clear message before launching child work rather than silently broadening Claude behavior.
- Queue, Auto, Guard, and Scheduler add the switch to their child argument vectors only when the operator supplied it. Their no-flag argument vectors do not contain the switch.
- Publish behavior is unchanged: default `pr`, explicit `branch`/`main`, `-NoPublish`, queue publish gates, inner-loop no-commit/no-push behavior, and issue/PR semantics are not changed.
- Focused tests prove the default workspace-sandbox path and the opt-in external-scratch path without touching `B:\sdk`, running Codex, calling GitHub, registering a scheduled task, or publishing.
- `AI_DISPATCH_AUTOMATION.md` includes the exact future operator command:
  `.\Invoke-AiDispatchAuto.ps1 -PublishMode pr -MaxAutonomousTasks 1 -MaxPlanRevisions 1 -MaxCorrectionRounds 2 -Executor codex -CodexExecutorExternalScratch`
- `Status.md`, `HANDOFF.md`, `change.md`, and `.ai/dispatch.tasks.md` record that ISSUE-340 is automation hygiene for future `B:\sdk` measurement dispatches and does not itself run a new measurement.

The Executor MUST halt without committing and request fresh guidance if any of the following occur:

- Implementing the option requires editing any forbidden path or adding files outside the MAY add surface.
- The only viable implementation would change default sandbox behavior, planner/reviewer sandbox behavior, publish behavior, queue publish gates, or scheduler registration state.
- The option cannot be limited to Codex execution, or it cannot fail fast for `-Executor claude`.
- Testing the option would require actually creating, deleting, or measuring under `B:\sdk` instead of inspecting hermetic argument vectors.
- PowerShell parser checks or focused Pester tests fail for reasons related to the change.
- Existing dirty tracked changes outside this task's allowed surface make it impossible to distinguish this dispatch's diff from unrelated work.
- The implementation reveals that Codex CLI no longer supports the intended external-scratch-capable sandbox mode; record the evidence and halt rather than inventing a partial workaround.

Keep the wording clear that this is for measurement tasks whose TASK packet explicitly authorizes `B:\sdk` scratch targets; it is not a general default for autonomous dispatch.
